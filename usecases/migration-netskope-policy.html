Netskope Policy Migration

A mature Netskope tenant encodes years of policy decisions: Real-time Protection rules, inline CASB activity and instance controls, DLP profiles, and a modest Cloud Firewall rulebase. This page translates each domain into Cato constructs through the Cato PS four-phase process — Export → Review & Map → Deploy (monitor-first) → Optimise — never lift-and-shift: rules are rationalised on usage evidence before anything is mapped, and EDM datasets are rebuilt from the source system, never imported.

Business objective

Policy translation is where migrations stall

Research-based page — not a delivery-verified runbook

This mapping is authored from Netskope and Cato documentation plus the Cato PS competitor-migration methodology — not from a completed, referenceable policy conversion. Review it with Cato Professional Services before using it with a customer, especially the DLP/EDM rebuild scope, the current SaaS Security API connector list, and app-by-app CASB activity coverage.

Connectivity moves in weeks — Sockets ship, tunnels come up, users get a new client. What stalls a Netskope migration is the policy estate: hundreds of Real-time Protection rules, DLP profiles nobody fully remembers commissioning, instance-aware CASB controls, and a steering-exception list that encodes years of operational archaeology. Translate it badly and the cutover either breaks the business (over-blocking) or silently drops controls (under-blocking).

Cato PS runs the policy workstream as a four-phase pipeline, applied per domain: Export the vendor policy (console plus REST API v2 where endpoints exist), Review & Map — rationalise on hit counts and usage evidence, discard dead, duplicate and shadowed rules, then map what survives to Cato constructs — Deploy everything monitor-first with no user impact, and Optimise: flip to enforcement rule by rule as events prove each control clean. Never lift-and-shift. The migration is the one chance to shed a decade of rule debt, and Cato's single-pass policy model rewards a smaller, cleaner rulebase.

Ask yourself

“Of your Real-time Protection rules and DLP profiles, which actually fired in the last 90 days — and who owns the ones that didn't?” Most estates enforce a small subset of what is configured. The firing subset is the migration scope; the rest is a rationalisation log, not a mapping task.

Domain 1 — FW rules

Cloud Firewall → Internet Firewall

Netskope Cloud Firewall is FWaaS for non-web egress: policies match users, groups and OUs against firewall app definitions (destination ports, protocols, IPs/FQDNs) with allow or block actions. In most estates this is the most modest domain — Netskope was usually bought for web, CASB and DLP, and Cloud Firewall only ever judged the traffic that steering delivered to NewEdge. Export: from the tenant console; REST API v2 exposes some but not all configuration, so plan for manual export and screenshots for the remainder.

Semantic gotchas

  • Coverage changes at cutover. Cloud Firewall only judged steered traffic; steering exceptions bypassed it entirely. Cato inspects all egress from Sockets and Clients, so hit patterns shift and quiet rules wake up — that is a feature, but only if you watch it in monitor before enforcing.
  • Default posture. Cato's Internet Firewall is a blocklist — an implicit final allow. If the customer ran default-deny for non-web egress, recreate that deliberately with an explicit catch-all block beneath the allows; never assume the posture travels with the rules.
  • Port-80/443 rules are usually dead on arrival. Cloud Firewall rules that duplicated web handling collapse into the web policy on Cato — the single engine owns both. Mark them discarded, don't map them twice.

Strategy: export, discard zero-hit and duplicated rules, map the survivors onto Internet Firewall rules with custom services, deploy with event tracking, watch hit counts against the Netskope baseline, then tighten.

Domain 2 — SWG policy

Real-time Protection web policy → Internet Firewall + Threat Prevention

Real-time Protection (RTP) policies are the heart of a Netskope web estate: top-down rules matching users/groups/OUs plus URL categories (predefined and custom), apps and Cloud Confidence Index (CCI) risk levels, with actions Allow, Block, Alert (silent log) and User Alert (a coaching page with justification capture) — alongside an SSL decryption policy and its do-not-decrypt bypass list. Export: policy list and custom categories from the console, partial coverage via REST API v2; notification templates need manual capture.

Netskope constructCato constructMigration note
RTP web rule (URL category → action)Internet Firewall rule on URL categoriesActions Allow / Block / Prompt in the same rulebase as FW rules
Custom URL categories & listsCustom categories in the CMARebuild these first — translated rules depend on them
CCI risk-level condition (0–100)App Catalog risk score (0–10, ACE)Map band-to-band; audit any rule pinned to an exact CCI score
User Alert / coaching templatePrompt actionWarn-and-continue only — no written-justification capture; set expectations
Alert (silent) actionAllow + event trackingNever translate Alert as Block — it changes user impact
Inline threat protection (AV / sandboxing)Anti-Malware / NGAM + IPSEnabled as Threat Prevention services, not per-rule profiles
SSL do-not-decrypt listTLS inspection bypass rulesThe single most valuable artefact in the export — seed Cato's bypasses from it

Semantic gotchas

  • Category taxonomies differ. Netskope and Cato URL categories are not 1:1. Review the mapping category-by-category with the customer — especially anything feeding an HR or regulatory control — rather than assuming name equivalence.
  • Inspection depth gates fidelity. Until TLS inspection is staged on, Cato matches at domain/SNI level — category blocks work, but URL-path distinctions and file-type controls need decryption. Netskope estates typically decrypted by default, so sequence TLS inspection early or accept a temporary fidelity gap.
  • Coaching depth. Netskope's User Alert captures justification text; Cato's Prompt is warn-and-continue. If written-justification workflows are a hard requirement, agree compensating controls (event follow-up via CMA/SIEM) — do not claim parity.
  • Bypassed traffic appears. Web traffic that steering exceptions never sent to RTP now hits Cato policy — expect novel events in monitor that are coverage gains, not regressions.

Strategy: rationalise on hit counts from tenant analytics, rebuild custom categories, deploy translated blocks as allow-plus-tracking, compare events against the Netskope baseline, then flip to Block — with Prompt standing in for former coaching rules.

Domain 3 — CASB policy

Inline CASB → App Control + Tenant Restrictions; API protection sequenced last

Netskope CASB is two halves. Inline: RTP rules keyed on app or app suite plus activity (upload, download, share, post, login) with instance awareness — corporate versus personal tenant. API Data Protection: out-of-band scanning of data at rest in sanctioned SaaS. Export: policy list and app-instance definitions from the console; REST API v2 covers part of it. The two halves migrate on different clocks — inline converts with the web policy, while API Data Protection keeps running untouched throughout the transition.

Netskope constructCato constructMigration note
App + activity rule (e.g. block upload to personal cloud storage)CASB Application Control rule (app + granular activity)Requires TLS inspection; validate activity coverage app-by-app
App instance / tenant conditionTenant Restrictions policy (header injection) + tenant awareness in App ControlDocumented for M365, Google Workspace, Slack, Dropbox — scope honestly beyond that list
App category / CCI conditionApp Catalog category / risk scoreRules migrate cleanly by category or risk band
Coaching on risky app activityPrompt actionWarn-and-continue; no justification capture
API Data Protection (data at rest)Cato SaaS Security APIDocumented connectors centre on M365 (SharePoint, OneDrive, Exchange) — keep Netskope API-CASB running during transition; retire it last

Semantic gotchas

  • Map intent, not construct. An instance-aware rule like “allow corporate OneDrive, block personal” is a tenant-control intent — it lands as a Tenant Restrictions rule plus App Control, not as a construct-for-construct twin. Write the mapping workbook in terms of what each rule is trying to stop.
  • Granularity honesty. Netskope's activity matrix is finer for some apps than Cato's App Control. Validate the customer's actual app/activity/instance matrix app-by-app in the pilot, and record any rule whose intent cannot be fully expressed — don't assert blanket equivalence.
  • Sequencing is the safety net. API Data Protection is out-of-band and conflict-free: it scans SaaS data at rest regardless of which vendor carries inline traffic. It stays licensed and running through the whole inline migration, and is retired last — only once Cato SaaS Security API coverage is confirmed for the customer's app list.
  • TLS inspection is a prerequisite. No decryption, no inline CASB — activity-level control needs to see inside the session. Stage TLS inspection before expecting App Control parity.

Strategy: list the inline CASB rules that actually fire, start from the Default Recommended CASB/DLP policy in the CMA, layer the mapped rules in monitor, and run a side-by-side comparison against Netskope events during the pilot before enforcing.

Domain 4 — DLP policy

DLP profiles → Data Control policy, EDM rebuilt from source

Netskope DLP profiles combine data identifiers (3,000+ advertised), ML classifiers, exact data matching (EDM), document fingerprinting and OCR; rules apply those profiles inline and via the API channel with severity and occurrence thresholds. Export: profile and rule definitions from the console — but EDM datasets cannot be exported: the uploaded records are hashed in a vendor-specific form. Every EDM dataset is rebuilt from the original system of record, which makes it the long-lead item of the whole policy workstream — start it in phase 1.

DLP profileDLP content profile (Data Control policy)350+ data types plus custom regex / keyword / file-label types
Predefined data identifiersCato data types by category (PCI, PII, PHI, credentials, source code…)Map the firing subset, not the licensed library
EDM datasetCato EDM — rebuilt from the system of recordNever exported or imported from Netskope; regenerate from HR / CRM / finance source data and agree a refresh cadence
Document fingerprintingIDM (indexed document matching)Re-index the source documents on Cato
OCR-based detectionCato OCROCR does not apply to EDM profiles — image-borne exact-match use cases need redesign
Inline DLP rule (profile + action)Data Control ruleRequires TLS inspection; deploy in monitor, flip per profile
API-channel DLP (data at rest)SaaS Security API DLPM365-centred connectors — same sequencing as API-CASB above

Semantic gotchas

  • EDM rebuild discipline. Engage the data owners (HR, finance, CRM) early — extracting a clean source dataset, agreeing fields and refresh cadence, and validating match behaviour takes longer than any rule mapping. This cannot be short-cut by “importing” anything from Netskope.
  • The OCR/EDM boundary. Netskope can exact-match inside images; on Cato, OCR and EDM are both available but not together. Where the customer relies on image-borne exact matches, redesign around OCR plus data-type profiles and record the residual gap honestly.
  • Compare hit parity, not datasheets. Netskope advertises a larger identifier library; most estates enforce a small firing subset. Run Cato DLP in monitor side-by-side during the pilot and compare hits on the customer's own traffic — that is the equivalence test that matters.
  • Expect a tuning cycle. Thresholds and severities translate approximately; plan a false-positive tuning loop per profile before any profile flips from monitor to block.

Strategy: rebuild profiles only for rules that fire, start from the Default Recommended CASB/DLP policy, regenerate EDM/IDM content from source systems, run monitor-mode parity during the pilot, then move to block profile by profile.

What changed in translation

  • Instance awareness became a tenant-control model. Netskope's per-rule instance condition (rule 1) has no construct-for-construct twin: the intent lands as a Tenant Restrictions rule allowing the corporate Google Workspace tenant by header injection, plus tenant awareness in the App Control rule — one Netskope rule became two Cato constructs, mapped by intent.
  • User Alert became Prompt — with less depth. Rules 1 and 3 keep their coaching behaviour, but Prompt is warn-and-continue: the written-justification capture in rule 3 has no Cato equivalent, so it goes in the mapping workbook as an intent not fully expressed.
  • The DLP profile was rebuilt, not moved. Rule 2's PCI profile maps onto Cato's predefined PCI data types; had it carried EDM identifiers, that dataset would be regenerated from the system of record — EDM content never exports from Netskope.
  • Activity granularity needs app-by-app validation. Upload translated cleanly here, but Netskope's activity matrix is finer for some apps — validate each translated activity in the pilot and record any that cannot be expressed, per the CASB gotchas above.
Cross-domain best practices

The rules that apply to every domain

🧭

Know Cato's default postures

The WAN Firewall is an allowlist (implicit final block); the Internet Firewall is a blocklist (implicit final allow). Every translated rule must land in the right posture — recreate default-deny egress deliberately, never by reflex.

🔐

Stage TLS inspection first

Decryption is the fidelity prerequisite for SWG activity rules, inline CASB and DLP. Enable it in planned stages and seed the bypass list from Netskope's SSL do-not-decrypt list — years of cert-pinning archaeology you should inherit, not rediscover.

👁️

Monitor, then block

Every translated control starts logging-only — allow-plus-tracking in the firewalls, monitor mode in CASB and DLP. Flip to enforcement rule by rule and profile by profile as the events prove each one clean.

🪪

Identity before policy

Almost every rule references a user group. Stand up Cato SCIM from the same IdP (Entra ID, Okta, OneLogin) alongside Netskope's — both provision independently — and confirm the groups exist in the CMA before the rules that need them deploy.

📊

Validate with events and hit counts

Compare hit counts pre- and post-migration. A translated rule that never fires is a removal candidate; a sudden hit spike usually means a category-taxonomy mismatch — investigate before enforcing.

🧹

Never lift-and-shift

Rationalise on usage evidence first: dead, duplicate and shadowed rules are discarded before mapping. The migration is the one chance to shed rule debt — a smaller rulebase is easier to prove correct in monitor.

The policy pipeline

One pipeline, four domains

Every domain — FW rules, SWG, CASB, DLP — moves through the same pipeline. The clean-up filter is what makes the migration safe: only rationalised, evidence-backed rules are mapped, everything deploys in monitor, and events close the loop before enforcement.
