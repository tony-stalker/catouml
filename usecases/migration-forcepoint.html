Forcepoint (Web Security & NGFW) to Cato

Forcepoint estates are proxy-heritage: PAC files pushed by GPO, Content Gateway appliances, a Stonesoft-heritage NGFW managed from SMC — and, in the middle of it, genuinely deep enterprise DLP. This use case walks the path onto the Cato SASE Cloud: retire explicit-proxy steering cohort by cohort, map web and firewall policy into one console, rationalise DLP honestly (including where a Forcepoint endpoint agent should stay), and decommission the appliance estate on the customer's contract timeline.

Forcepoint has publicly repositioned itself as an AI-powered data-security company. The moves are on the record: the Global Governments and Critical Infrastructure (G2CI) business was sold to TPG (completed October 2023, rebranded Everfox); Getvisibility (DSPM) was acquired in March 2025; and in April 2025 Forcepoint launched Forcepoint Data Security Cloud — with the former Forcepoint ONE SSE documentation now branded "Forcepoint Data Security Cloud | SSE".

None of that is an end-of-life announcement for the web security or NGFW products — do not claim one you cannot cite. The operational case stands on its own: PAC files and GPOs to maintain, proxy-embedded authentication, separate consoles for web, firewall and SSE (the ex-Bitglass Forcepoint ONE stack), and appliances to patch and refresh. Cato replaces the steering, web security, firewalling and remote access with one cloud platform and one console — while being honest that Forcepoint's endpoint DLP depth may deserve to stay.

Ask yourself

"Which Forcepoint products are actually deployed — cloud web security, Content Gateway, Forcepoint ONE, NGFW/SMC, DLP, email — when does each contract renew, and do you have any appliance EoS/EoL notices in hand?" The answers shape everything below.

Component mapping

What maps to what

Steering is the fundamental difference. Forcepoint web security is explicit — a PAC file or proxy setting decides what reaches it, and only proxied web ports are inspected. Cato is transparent — the Cato Client steers users and the Socket/IPsec tunnel steers sites, and every port and protocol is inspected at the PoP. That difference is also what makes co-existence clean (see below).

Be honest about DLP scope — it wins the room

Forcepoint enterprise DLP has real depth Cato's network DLP does not replicate: N-gram document fingerprinting, trainable ML classifiers, an OCR server with language packs, endpoint channels (USB, print, clipboard) and data-at-rest discovery. Do not contest it — reframe the scope. Cato enforces DLP inline on all traffic crossing the PoP, not just proxied web ports; the recommended design is a hybrid that keeps the Forcepoint endpoint agent wherever endpoint-channel mandates exist, with Cato device posture confirming the agent is running before granting access.

Recommended migration path

Five phases, each independently reversible

The sequence follows the Cato Professional Services methodology — discovery and design, parallel build and pilot, then phased cutover of users and sites, closing with optimise and decommission. Policy work follows the PS pattern: export → review & map → deploy → optimise.

  1. Discovery & design — inventory the steering layer and rationalise DLP first

    Document every PAC file and its GPO distribution points, policy-specific PAC URLs, WPAD, proxy settings hardcoded in apps and build tooling, and Content Gateway transparent-mode hooks (WCCP/policy routing) — the PAC estate is the rollback lever. Find proxy-embedded authentication: NTLM/Kerberos proxy auth and X-Authenticated-User headers consumed downstream — in Cato, identity comes from the Client and IdP integration, so every consumer needs an identity path. Enumerate egress-IP-pinned allowlists (SaaS tenant restrictions, conditional access, partner firewalls) for replacement with Cato allocated IPs.

    Rationalise DLP before mapping: age out unused policies, then classify what remains — dictionary/regex/predefined classifiers map directly to Cato data types; structured fingerprints re-implement as EDM; unstructured document fingerprints and ML classifiers have no direct equivalent — redesign around data types/EDM/MIP labels or explicitly retain Forcepoint for those policies. Decide the endpoint-DLP hybrid now, not at cutover.

  2. Pilot & parallel build — sockets alongside, certificate ahead of need

    Security → TLS Inspection

    Deploy Sockets/IPsec at pilot and hub sites in parallel with the existing NGFW/proxy egress — Cato carries traffic while Forcepoint still enforces web policy for everyone else. Roll the Cato CA certificate out fleet-wide before it is needed, then enable TLS Inspection gradually — test group first, expanding by URL category — with QUIC blocked via the Internet Firewall. Switch CASB discovery and DLP to monitor mode early and reconcile DLP monitor events against Forcepoint incidents before any enforcement.

  3. User cutover — remove the PAC cohort by cohort, Always-On Client in

    Access → Client Connectivity Policy

    Per cohort: remove or modify the PAC via GPO, enable the Cato Client with Always-On, and confirm nothing still steers web traffic to Forcepoint. Run weekly waves with tuning in between. Cato's Proxy Configuration Policy can centrally serve a transitional PAC for stragglers that still need one. Repoint SaaS tenant restrictions and partner allowlists to Cato allocated egress IPs before the owning cohort flips.

  4. Site cutover — retire NGFW and Content Gateway in sequence

    Network → Sites

    Cut each site's internet breakout to the Socket and Cato PoP; retire the local NGFW and Content Gateway once WAN and internet policy are confirmed on Cato, converting temporary routed-range designs to VLAN termination as firewalls come out. Retire the NGFW VPN gateway per region once its user population is on Cato ZTNA — keep it dark-standby until sign-off.

  5. Optimise & decommission — aligned to contract end-dates

    Monitor → Events

    Event-driven tuning across firewall, TLS, CASB and DLP; convert agreed DLP policies from monitor to block in stages. Decommission the Forcepoint cloud tenant, Content Gateway/V-Series appliances, SMC and NGFW estate and VPN gateways; reclaim GPO objects and PAC hosting. Forcepoint licensing is non-cancellable with reinstatement fees for lapsed support — so time non-renewal to the last cutover wave and align the plan to the contract end-date, not vice versa. Re-evaluate the DLP hybrid: if endpoint-channel and discovery mandates persist, keep a deliberately scoped endpoint-DLP licence, with Cato enforcing everything in motion.

Co-existence & rollback

Two steering planes that don't collide

This pairing co-exists unusually cleanly. Forcepoint's steering is explicit — the PAC decides what reaches it; Cato's is transparent — the Client and Socket own the path. The two planes are independent, so both platforms run in parallel without conflict: un-migrated cohorts keep the PAC pointing at Forcepoint (cloud or Content Gateway) while migrated cohorts steer transparently to the Cato PoP, where Internet Firewall, SWG, TLS and DLP policy now apply.

Explicit — PAC → Forcepoint (interim) Transparent — Cato Client / Socket
The two steering planes are independent: cohort A's PAC still sends web traffic to Forcepoint while cohort B steers transparently to the Cato PoP. Exactly one platform owns TLS inspection per cohort, and the Forcepoint DLP endpoint agent stays for its exclusive channels.

One TLS inspection owner per cohort — never decrypt the same flow twice. While a cohort's web traffic still terminates at Forcepoint, bypass Cato TLS Inspection for that cohort (or scope Cato TLSi to migrated cohorts only), and flip inspection ownership at cutover. Rebuild the certificate-pinned app bypass list on the Cato side from the Forcepoint bypass list — do not assume the two are identical.

Independent steering planes

Explicit PAC steering and transparent Client/Socket steering never fight over the same flow — both platforms run in parallel for as long as the waves need.

⏪

Rollback is a config event, not a project

Reinstate the PAC/GPO for the affected cohort, relax Always-On for that group, and traffic returns to Forcepoint at the next policy refresh.

🧊

Policy-frozen fallback

Keep the Forcepoint tenant and appliances licensed and policy-frozen until final sign-off — and freeze parallel edits on both platforms during waves to avoid drift.

🛡️

No protection gap on data

Cato DLP runs in monitor then block on data in motion, while the Forcepoint endpoint agent keeps covering USB, print and clipboard throughout.

Gotchas & objection handling

Where this migration bites — and what to say

"Forcepoint DLP is deeper than Cato DLP"

Partly true — don't contest it, reframe scope. Map each actually used policy (most estates run far fewer than they own): structured fingerprints → EDM; unstructured fingerprints and ML classifiers → redesign or retain hybrid. Test Cato's OCR limits (image formats, 10 KB–50 MB, no OCR with EDM profiles) against real policies in the pilot — never assert them.

Endpoint DLP channels

Cato enforces DLP at the PoP on traffic in motion — it is not an endpoint DLP agent. Regulated industries often have hard USB/print/clipboard mandates: lead with the hybrid design, not a swap, and use Cato device posture to require the endpoint agent before granting access.

Proxy-embedded authentication

NTLM/Kerberos proxy auth and X-Authenticated-User headers disappear when the proxy does. Identity moves to the Cato Client + IdP (SCIM/SSO) — anything downstream that reads proxy headers needs a redesign in phase 1, not a mapping.

PAC stragglers & hardcoded proxies

Apps and build tooling with hardcoded proxy settings keep sending traffic to a proxy that no longer exists. Sweep for them in discovery; Cato's Proxy Configuration Policy can serve a transitional PAC — the final state is no PAC at all.

SaaS & partners pinned to proxy egress IPs

Cutover silently breaks tenant restrictions, conditional access and partner allowlists keyed to Forcepoint egress ranges. Enumerate them in phase 1 and move each to Cato allocated egress IPs (per-app egress rules) before the owning cohort flips.

Quota / confirm browsing actions

Websense-era time-quota browsing has no like-for-like Cato action. Cato Prompt (warn-and-continue) covers the common intent — validate policy-by-policy rather than promising parity.

"Forcepoint says nothing is EOL"

Largely true for the cloud and DLP products — the pressure is strategic, not contractual: the data-security pivot, G2CI divested to Everfox, and console/agent fragmentation across ONE and the legacy stacks. Never claim an EOL you cannot cite; a model-level NGFW EoS notice the customer holds is the concrete trigger.

Email security is out of scope

Cato does not replace Forcepoint Email Security — scope it out explicitly and pair with M365-native or a dedicated vendor. Regulated customers should also map DLP incident export (Cato events → SIEM/SOAR) early and run both incident streams in parallel during monitor mode to build auditor confidence.
