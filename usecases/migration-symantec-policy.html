Symantec Policy Migration

A Symantec estate is a proxy estate: up to twenty years of CPL spread across VPM-generated and hand-written Local files, WSS/Cloud SWG portal policy, CloudSOC CASB controls and a genuinely deep Symantec DLP suite. This page works through the translation of each domain into the Cato Management Application using the Professional Services pattern — export, review & map, deploy monitor-first, optimise — with the CPL corpus rationalised rather than ported, proxy authentication re-designed around SCIM identity, and an honest hybrid where Symantec DLP's fingerprinting and endpoint depth should stay.

Business objective

Policy translation is where migrations stall

Research-authored page — validate before deployment

This page is authored from public research, not a delivery-verified runbook. Review the mapping and sequencing with Cato Professional Services before using it to plan a customer engagement — and verify every Symantec-side export path against the customer's actual SGOS versions, UPE topology and product mix.

Connectivity moves fast; policy is the long tail. Sockets tunnel up in days — what stalls a Symantec displacement is translating decades of accumulated policy across four consoles (VPM/Management Center, the WSS portal, CloudSOC and DLP Enforce) into one coherent CMA rulebase without losing the controls that matter or importing the clutter that doesn't. The Cato PS pattern for this is a four-phase pipeline: EXPORT → REVIEW & MAP → DEPLOY (monitor-first) → OPTIMISE.

The cardinal rule: never lift-and-shift — and for CPL there is no other option, because no automated CPL-to-Cato converter exists. That is good news in disguise: most CPL volume is dead layers and workarounds for proxy behaviour — auth exemptions, pinned-certificate SSL bypasses, WCCP quirks — that simply cease to exist in a transparent Client/Socket model. Pull hit data first (Reporter reports and access logs, while they still exist), rationalise against it, and expect the mapped set to be a fraction of the CPL line count. That reduction is the value story, not a loss.

Ask yourself

"Roughly how many CPL layers and rules are live today, when were they last audited — and can you show me hit counts per layer from Reporter or the access logs?" If Reporter has already been switched off, the export phase starts by mining whatever access-log archive survives.

Domain 1 — FW rules

Proxy-era network control → Cato WAN & Internet Firewall

ProxySG is a proxy, not a firewall — yet in a long-lived estate a surprising amount of network control has accumulated inside it: CPL conditions keyed on client addresses and ports, static bypass lists, WSS network policies covering non-web ports, WCCP redirect ACLs on adjacent routers, and proxy-forwarding chains from branch appliances to the cloud. The first job is separating genuine L3/4 intent — which maps to the Cato Internet and WAN Firewalls — from steering plumbing, which retires with the proxy. Be clear with the customer: most CPL is web policy, not firewall policy, and it is handled in Domain 2.

Semantic gotcha — two default postures, and very little true firewall

Cato splits the estate: the WAN Firewall is an allowlist (implicit final block) and the Internet Firewall is a blocklist (implicit final allow, hardened with Cato's recommended block baseline). A CPL rule is not firewall policy just because it mentions an IP address — the test is intent: does it control network reachability (firewall) or web content (SWG)? Misclassifying web policy as firewall rules recreates the clutter you are migrating away from.

Strategy: expect this to be the smallest domain, and say so — it reframes the conversation from "port our firewall" to "deploy Cato's recommended Internet Firewall baseline plus your genuine deltas". Extract the L3/4 subset from the CPL export with hit evidence, rebuild the objects in CMA, and let the WCCP, bypass and forwarding plumbing retire without replacement.

Domain 2 — SWG policy (the deep one)

CPL web policy → Internet Firewall & App Control

This is where the estate's weight sits. Policy lives in the VPM policy file (generated CPL) plus hand-written CPL in the Local — and sometimes Central and Forward — policy files, evaluated in order-dependent layers. Export all of them from each appliance; on UPE estates, export from the reference Edge SWG device via Management Center and treat that corpus as the single source of truth for the appliances and the Cloud SWG tenant. Non-UPE cloud policy is captured separately from the WSS portal.

Then triage before you map: classify every layer as business intent (maps to Cato), proxy plumbing (auth exemptions, pinned-cert SSL bypasses, WCCP workarounds — dies with the proxy) or dead (delete). The evidence is rule hit data from Reporter and the access logs — and with on-prem Reporter EOL since March 2025, that evidence base is on borrowed time. Pull the reports and archive the access logs in week one of discovery, not at decommission.

Semantic gotcha — the proxy authenticated, Cato identifies

Every IWA estate carries a shadow policy of authentication mechanics: Kerberos SPNs tied to the appliance FQDN, virtual URLs for transparent auth, NTLM fallback rules and guest exemptions. None of it maps — identity moves to the Cato Client plus SCIM/SSO provisioning, and user- and group-based web rules then reference IdP groups directly. Provision the SCIM groups before importing a single user-based rule, and treat any system that consumed forwarded user headers as a redesign item, not a migration line.

Strategy: start from Cato's recommended block baseline, then layer the customer's genuine deltas — custom categories, coach→Prompt conversions, documented exceptions — on top. Run the result in monitor alongside the remaining ProxySG/WSS reporting and compare category hit volumes before enforcing; taxonomy deltas show up as event noise long before a user complains. If the mapped rulebase is not dramatically smaller than the CPL line count, the review phase didn't happen.

Domain 3 — CASB policy

CloudSOC → Cato CASB & App Control

The CloudSOC estate is usually the smallest domain: Audit shadow-IT discovery fed by SpanVA log-collector appliances, Securlets providing API-mode control over sanctioned apps, inline access and activity policies via the Cloud SWG integration, and DLP-paired content inspection. There is no clean bulk export — capture the policy intent from the console: the sanctioned-app list, allowed activities per app, and which policies pair with DLP.

Audit shadow-IT reports (SpanVA collectors)Cloud Apps Dashboard, App Analytics, app risk scoresSpanVA appliances retire — Cato sees the traffic at the PoP, so discovery needs no log collectors. Turn it on from day one in monitor.
Securlets — API mode per sanctioned appCato API CASB per appCoverage differs per SaaS app on both platforms — validate app-by-app during design, never assume parity.

DLP-paired CASB policiesDLP rules in the same CMA rulebaseSingle engine and console — no cross-module pairing to maintain.
Cloud SWG egress-IP allowlists (published ranges)Allocated egress IPs + per-app egress rulesSaaS tenant restrictions and partner allowlists keyed to Broadcom's ranges break silently — repoint before the owning cohort flips.

Gotchas and strategy: decommission CloudSOC only after API-CASB parity is validated for every sanctioned app, and enforce inline CASB granularity only once TLS Inspection covers the cohort — inline activity control depends on decrypted traffic. Because the domain is small, map it late, run discovery in monitor from the start to validate the sanctioned-app list, and treat the egress-IP repointing as a hard prerequisite in the wave plan.

Domain 4 — DLP policy

Symantec DLP → Cato DLP, detection tech by detection tech

This is the domain that decides credibility. Symantec DLP is genuine enterprise depth — strategically retained under Broadcom, current at 16.1, and rarely the first thing a customer leaves. Its network rules reach Cato via ICAP from the proxy or the Cloud Detection Service; its policies mix four detection technologies with very different migration outcomes. Rationalise before you map: pull the policy list and the Enforce incident history, age out every policy that has never raised a triaged incident, then classify what remains by detection tech.

The honest hybrid — recommend it, don't concede it

Where regulation mandates endpoint DLP channels or data-at-rest discovery — common in financial services, health and legal — the recommended design is a hybrid: Cato DLP enforces everything in motion at the PoP, while a deliberately scoped Symantec DLP footprint keeps the endpoint agent and Discover, with a Cato posture check confirming the agent is running. That is an architecture decision made in the review phase — not a gap discovered at cutover — and it is re-evaluated at each Broadcom renewal.

Strategy: classify every surviving policy into three buckets — direct map (described content → Cato data types), rebuild (EDM, regenerated from source data) and redesign-or-retain (IDM and VML) — and record the decision per policy. Deploy all mapped content profiles in monitor mode and reconcile Cato DLP findings against the Symantec Enforce incident queue over a full business cycle before any cutover. Only then convert to block, in stages, highest-confidence policies first, keeping the endpoint agent live throughout for its exclusive channels.

Worked example

Before → after: CPL translated to intent

Illustrative extract — not a console export

The CPL below and the CMA table that follows were composed for this page to show the translation pattern. They are not screenshots of either console, and the CPL is a simplified sketch of what a real Local policy file looks like. All names are generic placeholders — though the screenshot beneath the after-table is the real CMA console, captured from a live demonstration environment.

The mapping tables above say what converts to what — this is what the triage looks like on a few lines of policy. One Local-file extract containing four CPL constructs: a custom-category definition, a group-scoped deny, an SSL-intercept exception for online banking, and a legacy user-agent auth exemption nobody has dared touch.

Before — ProxySG CPL extract (illustrative)

; Local policy file — extract
define condition Unapproved_FileSharing
  url.domain=fileshare.example url.domain=bulkdrop.example
end

<Proxy "Web-Access-Standard">
  group="Standard-Users" condition=Unapproved_FileSharing deny
  ; legacy Java updater could not complete NTLM — Reporter shows zero hits since audit
  request.header.User-Agent="Java/1.6" authenticate(no)

<SSL-Intercept "Intercept-Exceptions">
  category="Financial Services" ssl.forward_proxy(no)

After — Cato CMA (illustrative)

Only the intent crosses, and it lands in two policies: the file-sharing deny becomes an Internet Firewall rule, the intercept exception is rebuilt in the TLS Inspection policy — and the dead rule goes to the discard pile with the hit data as evidence.

1Block-Unapproved-FileSharingStandard-Users (SCIM group)Custom category: Unapproved File SharingBlockEvent

Notice what has no row at all: the authenticate(no) exemption mechanics, the IWA realm behind group=, and any proxy-chaining or virtual-URL redirects around this policy. That is plumbing — nothing challenges the browser any more, so there is nothing to migrate.

What changed in translation

  • Order-dependent CPL became declarative criteria — the define condition plus its layer rule collapse into one Internet Firewall rule whose criteria (group, custom category, action) travel together; there is no layer evaluation order left to reason about.
  • The triage did the real work — of four CPL constructs, two carried business intent, one was dead on the Reporter evidence, and the auth exemption was proxy plumbing. Only the intent crossed; that ratio is typical, and it is the value story.
  • IWA/BCAAA gave way to SCIM identity — group= no longer resolves through an IWA realm, BCAAA server or Kerberos SPN: Standard-Users is a SCIM-provisioned IdP group referenced directly in the rule, provisioned before the rule was imported.
  • The SSL-intercept exception changed policies — ssl.forward_proxy(no) does not travel with any web rule; it is deliberately rebuilt as a TLS Inspection bypass rule, verified against current banking-app behaviour rather than copied from the old exception list.
Cross-domain best practices

Rules that hold across all four domains

🧭

Two default postures — learn them first

Cato's WAN Firewall is an allowlist (implicit block); the Internet Firewall is a blocklist (implicit allow, hardened by the recommended baseline). Every mapping decision flows from which side of that line a rule lands on.

🔐

Staged TLS inspection is the fidelity prerequisite

SWG granularity, inline CASB and DLP all depend on decrypted traffic. Certificate fleet-wide first, QUIC blocked, test group, expand by category — and exactly one inspection owner per cohort while ProxySG, SSLV or WSS still decrypts for un-migrated users.

📡

Monitor-then-block, everywhere

Every migrated rule ships watching before it ships blocking: monitor mode for Internet Firewall deltas, CASB and DLP alike. Enforcement is earned with telemetry, never assumed from a mapping workbook.

🪪

SCIM-first identity

IWA realms, BCAAA, Kerberos SPNs and forwarded user headers all die with the proxy. Provision IdP/SCIM groups before importing any user-based rule, and give every consumer of proxy auth an identity path — not a mapping.

📊

Validate with events and hit counts

The acceptance test is quantitative: compare Reporter and access-log baselines pre-migration with Cato event and rule-hit data post-deploy. Parity in the numbers, not in the rule count.

🗑️

Rationalisation is the deliverable

The point of export → review is a smaller, explainable rulebase. If the CMA policy set is anywhere near the CPL line count, plumbing and dead layers were imported — go back to the review phase.
