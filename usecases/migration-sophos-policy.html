Sophos Policy Migration

Swapping an XGS for a Socket is a routing exercise — the SFOS policy estate is where the project earns its keep. This page works domain by domain through the Sophos Firewall policy stack: firewall rules with their attached web, application-control, IPS and anti-virus policies, the decoupled NAT rule table, and the SSL/TLS inspection rules — showing what each construct becomes in the Cato Management Application, the semantic traps in between, and the four-phase pipeline that rationalises the rulebase on usage evidence before anything is translated. CASB and DLP have no real SFOS ancestor, so they are built greenfield from Cato best practice — the upgrade, not the gap.

Business objective

Why policy translation is the long pole

Sockets ship, subnets move, sites cut over — and then the project meets years of accumulated SFOS policy: per-appliance rule tables, near-identical web-policy variants per user population, a NAT table full of forgotten DNATs, and rules conditioned on things that will not exist after the firewall goes. Translating that estate badly does more damage than translating it slowly: lift-and-shift faithfully reproduces every dead rule, every any-any exception and every policy carve-out on a brand-new platform, and the customer inherits their old technical debt with a new logo on it.

The Cato Professional Services approach is a four-phase pipeline, applied per policy domain: EXPORT the configuration and its usage evidence → REVIEW & MAP — rationalise on per-rule usage, discard dead weight, then map surviving intent to Cato constructs → DEPLOY monitor-first, so every translated rule earns evidence before it can hurt anyone → OPTIMISE against events and hit counts until the policy is demonstrably right. Review-and-map, not lift-and-shift: per-box rule bases collapse into one global policy, and the clean-up is what makes that mapping tractable.

This page covers the policy workstream only. The network side — parallel Socket deployment, the hub interconnect, SD-RED swap-outs, remote-access cohorts and the Sophos-endpoint-stays framing — is the companion page, Sophos (XGS Firewall & Central) to Cato.

“How many of your SFOS firewall rules have passed no traffic in the last 90 days — and how many of the rest are conditioned on a Security Heartbeat status that won't exist once the firewall is gone?”

Domain 1 — FW rules

SFOS firewall rules & the NAT table

The construct. SFOS firewall rules are ordered, zone-based rules in a per-appliance table: source/destination zones (LAN, WAN, DMZ, VPN, Wi-Fi), IP hosts and host groups, services, identity (users and groups), schedules — with security delivered as policies attached per rule: a web policy, an application filter policy, an IPS policy, anti-virus scanning and zero-day protection. Inspection happens only where a policy is attached. Since SFOS 18, NAT lives in its own decoupled rule table — source NAT/masquerading for egress, DNAT server-access rules for published services — rather than inside each firewall rule. Synchronized Security adds a Sophos twist: rules can be conditioned on the endpoint's Security Heartbeat status.

Export. The full backup restores a firewall — it does not feed a mapping workbook. The parseable artefacts are the XML configuration export and the SFOS XML API: 

# Per appliance — full backup is an (encrypted) restore artefact, not a mapping input
Backup & firmware → Backup & restore

# The parseable rulebase: XML configuration export, or the XML API for scripted pulls
Backup & firmware → Import export → Export full configuration

# Usage evidence: per-rule traffic counters in the rule table + Central Firewall Reporting

Semantic gotchas. One SFOS rule table becomes layers: a single rule with its attached policies decomposes into a firewall rule, account-level IPS/NGAM settings and a TLS inspection rule — agree the layer mapping before translating rule one. Zone semantics have no equivalent; recover the intent (which segment reaches what) rather than transliterating zone pairs. Traffic that escaped inspection on SFOS because no policy was attached will be inspected by Cato's single-pass engine — expect new detections in the monitor phase and triage them before enabling block. And heartbeat-conditioned rules are the trap that bites late: they silently encode a firewall↔endpoint dependency that disappears with the XGS, so inventory them in discovery, not after the objection lands.

Strategy. Pull the XML export per appliance; retire zero-traffic rules and collapse duplicate per-box objects against the usage evidence; push the cleaned object model through the API or Terraform first, then build the rule layers on top of it; deploy new WAN Firewall and Internet Firewall rules monitor-first and let events prove them before enforcement. Treat the NAT table as its own mini-workstream — most of it dissolves, and what survives (published services) moves on scheduled change windows.

Domain 2 — SWG policy

Web policies, application control & SSL/TLS inspection

The construct. SFOS web filtering lives in web policies attached per firewall rule: ordered rules matching users/groups against activities — web categories, URL groups, file types — with Allow, Warn or Block actions and time constraints, usually accumulated as several near-identical policy variants, one per user population. Application control is a separate attachment: application filter policies matching apps by category, risk and characteristics, with Synchronized App Control using the endpoint agent to classify traffic the engine cannot identify. Since SFOS 18, SSL/TLS inspection is its own rule table with decryption profiles, sized against the appliance's inspection throughput. Sophos DNS Protection enforces similar categories at the resolver.

Export. All of it travels in the same XML configuration export as the firewall rules — web policies, URL groups, application filter policies and the SSL/TLS inspection rules — with Central Firewall Reporting as the evidence base for what each category action actually affects.

Semantic gotchas. The taxonomy mismatch is the headline risk: a Sophos category and a similarly-named Cato category are not guaranteed to cover the same sites, so a mechanical name-for-name mapping silently changes what is blocked. Fidelity is the second: without TLS inspection both platforms act on domain/SNI only, so stage TLS inspection before judging SWG parity — and note that on SFOS, inspection scope was often set by appliance headroom rather than policy intent. Sophos' own XGS marketing leads with 4.8–5× TLS-inspection throughput gains over XG precisely because inspection throughput is what runs out; on Cato the sizing exercise does not exist, so the migrated TLS policy should encode what the customer wants decrypted, not what the box could afford. Finally, policy sprawl should die here: collapse the near-identical web-policy variants into a small number of group-scoped rule tiers instead of recreating each one.

Strategy. Build a category translation sheet mapping each Sophos category-action to a Cato category and action by intent; consolidate URL groups into custom categories; deploy the blocked set in monitor mode and compare events against Central Firewall Reporting; flip to block per category once the evidence matches expectations, using Prompt where the customer used Warn.

Domain 3 — CASB policy · greenfield

CASB: not a translation — an upgrade

There is usually nothing to export. Sophos Firewall has no inline CASB: SaaS control in an SFOS estate — where it exists at all — is a few web categories and application filter rules doing a job they were never designed for, with cloud-application visibility limited to usage reporting. Frame this honestly in the room: this is not a migration gap, it is the upgrade the customer is buying. The CASB policy is built greenfield from Cato best practice, not translated from a rulebase that does not exist.

What the estate does contribute: application usage data from the firewall's reporting seeds the sanctioned-app conversation — which SaaS the business already lives in — and the category worksheet from the SWG domain flags any informal SaaS blocks worth preserving as proper CASB rules.

In the Sophos estateCato constructWhat changes
App/web reporting on the firewall and in Central App Analytics + Cloud Apps Dashboard Continuous shadow-IT discovery with risk scores, covering sites and remote users alike
Web/app categories doubling as SaaS control CASB application rules with granular activity controls Allow the app but govern the activity — upload, download, share — instead of a binary block
Nothing — no app risk model Application catalogue with risk scoring Sanctioned / tolerated / unsanctioned decisions made on evidence, per app
Nothing — no tenant awareness Tenant restrictions on sanctioned apps Corporate tenant allowed, personal instances of the same app controlled

Build sequence (best practice). Stage TLS inspection first — CASB activity-level control depends on it. Run shadow-IT discovery over a representative business cycle in App Analytics and the Cloud Apps Dashboard; classify the estate into sanctioned / tolerated / unsanctioned with the risk scores as evidence; write CASB rules in monitor mode; review the hits with the business owners; then enforce, strictest-risk apps first.

Position it as the upgrade it is

“Day one on Cato you will see every SaaS app in use, scored for risk — visibility the firewall estate has never had. We turn that into control at your pace: monitor first, then enforce.” That is a selling point, not a migration risk.

Domain 4 — DLP policy · greenfield

DLP: build from the data, not from a rulebase

Again, nothing to export. Sophos Firewall carries no network DLP rulebase: the nearest constructs are file-type blocks inside web policies — upload hygiene, not data protection — and any email data-control rules live in the email stack, which this migration does not touch. Like CASB, this is a greenfield build, and the input is not a config file but a conversation: a short data-protection workshop with the compliance or security owner on what data matters (cardholder data, PII, credentials, source code), where it lives, and who legitimately moves it.

In the Sophos estateCato constructWhat changes
File-type blocks in web policies DLP file-type controls inside content-aware rules File type becomes one condition among many, not the whole policy
Nothing — no content awareness on the wire DLP profiles with predefined data types (PCI, PII, credentials, source code) Start from the predefined types — do not hand-build patterns on day one
Nothing — no SaaS-aware data policy DLP rules scoped by app, category, user and direction “Finance may upload PCI to the sanctioned tenant; nobody uploads it anywhere else”
Nothing — no evidence trail Data Protection Dashboard + DLP events Every match attributed to user, app and data type — the audit trail the estate never had

Build sequence (best practice). TLS inspection staged first — DLP cannot inspect what it cannot see. Enable the predefined data types that match the workshop outcomes in monitor mode; review matches on the Data Protection Dashboard and tune false positives with the data owners; enforce on the highest-confidence rules first (cardholder data before “possible PII”), and widen coverage iteratively. Day-one blocking is how DLP projects die — monitor-mode evidence is how they succeed.

Before → after: an SFOS rulebase translated

Illustrative only

The tables below are composed to show the translation pattern — they are not console exports, and a real rulebase runs to hundreds of rows — though the screenshot after the CMA table is the real console from a live demonstration environment. Every surviving row still travels the full pipeline: rationalise on usage, map, deploy monitor-first, then enforce on evidence.

Before — SFOS firewall rule table (illustrative)

1 Corp-Web-Access LAN → WAN Corp-Networks Any Accept Corp-Web policy · Block-High-Risk-Apps · IPS LAN-to-WAN · AV scanning

Alongside the rule table sits the NAT table: a masquerade rule NATing all egress to the WAN interface, and a DNAT server-access rule publishing an internal web server. Neither is “translated” in place — the masquerade dissolves into PoP egress NAT, which is a platform property, and the published service moves to remote port forwarding on an allocated IP in its own change window, after checking anyone still uses it.

After — Cato CMA (illustrative)

#NameSourceApp/CategoryServiceActionTrack
1 Guest Wi-Fi — allow web Guest-Network Any HTTP · HTTPS · DNS Allow Event
2 Guest Wi-Fi — block everything else Guest-Network Any Any Monitor → Block Event
3 Corp — block restricted categories Corp-Networks Categories mapped by intent from Corp-Web policy Any Monitor → Block Event

Several Before rows have no After row — deliberately. The IPS and AV attachments on rule 1 are no longer per-rule properties: they become the account-level IPS and Next-Gen Anti-Malware layers of the single-pass engine, applied to everything it sees — including the guest network that SFOS left uninspected. There is no “Corp-Web-Access” allow rule either — the Internet Firewall's default-allow posture does that job, so what survives of rule 1 is its web-policy intent, as rule 3, with the Block-High-Risk-Apps filter re-expressed as app-aware rules. Rule 3 exits the pipeline at REVIEW & MAP on its usage evidence, and rule 0 — the default drop — is a posture decision, not a rule: the WAN Firewall is implicit-block already, and if the customer wants allowlist egress it becomes an explicit final block rule.

What changed in translation

  • Attached policies became layers. The web + app filter + IPS + AV stack attached to rule 1 decomposed into an Internet Firewall category rule plus account-level IPS and NGAM — separate global policy layers of the single-pass engine, with the old per-rule carve-outs re-expressed as explicit, auditable exceptions.
  • Zones became scopes. LAN → WAN has no equivalent construct — the intent (which segment reaches the internet, under what constraints) is recovered as source scopes in one global rulebase, as the guest rules show.
  • The NAT table dissolved. Masquerade rules are a platform property at the PoP; only the published services survive, re-homed to remote port forwarding on their own change windows.
  • Usage evidence decided what never made the journey. The zero-traffic legacy rule was discarded at review rather than translated — the discard pile, and the posture decision that replaces the default drop, are outputs of the migration as much as the rules that survive.
Cross-domain practices

Rules that hold across all four domains

🧭

Respect the default postures

Cato's WAN Firewall is an allowlist (implicit block); the Internet Firewall is a blocklist (implicit allow). SFOS drops by default everywhere — so egress posture is a decision, not a translation. If the customer wants allowlist egress, add an explicit final block rule; don't discover the difference in production.

🔐

Stage TLS inspection early

Without decryption, SWG acts on domain/SNI only and CASB activity control and DLP content inspection cannot see payloads. Phased TLS inspection — by category and pilot group, certificates deployed in advance — is the fidelity prerequisite for three of the four domains, and no longer an appliance-sizing question.

🟠

Monitor, then block

Every translated rule, IPS/NGAM protection, CASB control and DLP rule starts in monitor. Enforcement is earned with event evidence, per rule — the pipeline's amber-to-green transition, never a big-bang flip.

🪪

Sequence identity first

Group-scoped rules are only as good as the groups. Provision Entra ID SSO and SCIM sync before policy deployment, and map the AD users and groups referenced in SFOS rules to the synced directory groups — name drift here breaks rules silently.

🧱

Objects before rules

Translate and dedupe the host/service object model first — scriptable from the XML export via the GraphQL API or Terraform provider — then build rules on the clean object base. Per-box object copies recreate the sprawl you just removed.

📊

Validate with events & hit counts

The evidence loop closes in Monitor → Events: compare each translated rule's hits against its SFOS ancestor's traffic counters, investigate the deltas, and treat zero-hit rules after a full business cycle as candidates to delete — on the new platform as ruthlessly as on the old.

The policy pipeline

EXPORT → REVIEW & MAP → DEPLOY → OPTIMISE

The same pipeline runs for each domain, offset in time — firewall first, SWG behind it, CASB and DLP once TLS inspection and discovery data are in place. The two rulebase domains enter with an XML export; the two greenfield domains enter with discovery inputs — usage data and the data-protection workshop — and skip translation, not the monitor-first discipline. Dead rules leave the pipeline at review and are never translated; everything that survives deploys in monitor and is promoted to enforcement on evidence.

Monitor first — log, don't block Enforce — promoted on evidence Discarded — dead rules never migrate
The four Sophos policy domains are exported once — the greenfield pair carrying discovery inputs rather than rulebases — rationalised against usage evidence, where dead rules and any-any debt exit the pipeline, and only the surviving intent is mapped to CMA constructs. Every mapped rule deploys in monitor (amber) and is promoted to enforcement (green) when events and hit counts prove it behaves as intended.
