Migration Methodology & Co-existence Patterns

Every SASE migration lives or dies on one question: how do the legacy network and Cato co-exist while sites move? This page is the Professional Services playbook — structured discovery, big-bang vs phased, parallel connectivity vs interconnect hubs, routed ranges vs BGP handoff, and the six multi-region patterns that keep routing symmetric while both networks run side by side.

Business objective

Why this matters

Prospects rarely doubt that Cato works — they doubt that they can get there without an outage, a stalled project, or a weekend of heroics. Being able to walk a customer through a proven migration methodology is often the difference between a stalled evaluation and a signed order: it shows the path from the current estate to Cato is engineered, reversible and boring — in the best possible way.

The method is always the same: discover the environment properly, choose the migration approach deliberately, and design the co-existence pattern before the first site moves. Everything below comes from the Cato Professional Services common-migration-strategies playbook.

Phase 1 — Discovery & assessment

No design before discovery

Professional Services opens every engagement with a project kickoff and a structured network and security discovery. The interactive sessions and the questionnaire exist for one reason: the right co-existence pattern is dictated by facts about the environment — it cannot be chosen from a template.

Ask yourself

“While both networks are running, where does the routing decision between legacy and Cato live — an L3 switch, the existing firewall, or the Cato socket? And if a migrated subnet misbehaves at 2am, is rollback a route change or a site visit?”

Phase 2 — Choose the approach

Big Bang vs Phased Migration

There are two macro approaches to network migration: Big Bang — cut the whole estate over to Cato in a single operation — or a more gradual phased migration using interconnecting sites and parallel sockets.

Big Bang LAN cutover

Physical cable, VLAN or route cutover in one window. Suitable for fewer sites or simple connectivity; validation testing is still required.

  • + Shorter project timeline
  • + Potential for reduced costs*
  • − Higher exposure to risk
  • − Resource-intensive during the cutover window
  • − Complex rollback

Phased Migration Interconnect / parallel

Interconnecting site or parallel sockets; both networks run side by side while sites move in waves.

  • + More gradual, less risk exposure
  • + Parallel testing and validation is possible
  • + Issues are uncovered while isolated to migrated sites
  • − Longer project timeline
  • − Potential for increased costs from running both networks*

* Depending on approach, legacy service agreements and MSA contracts.

Phase 3 — Technical co-existence

Parallel side-by-side connectivity

Parallel side-by-side connectivity is the foundation for every migration strategy — big bang, parallel connectivity across all sites, or a central hub-site interconnect. The design question is always the same: which device makes the routing decision between the legacy and Cato networks? Default-gateway placement, LAN architecture (L2 vs L3), VLAN termination point, the Cato site type selected and plans for the existing firewall drive the choice between three patterns:

L3 switch as DG

Where a layer-3 switch terminates the VLANs, it makes the route decision. Configure a transit VLAN (optional), statically route migrated ranges towards Cato — or propagate them dynamically via BGP.

Legacy firewall as DG

Where the existing firewall or router terminates the local VLANs, it controls Cato routing. Establish a transit VLAN between the socket LAN and the firewall; route migrated ranges statically or via BGP.

Cato socket as DG

Where the socket should be the default gateway from day one, use a BGP handoff on LAN or Alt-WAN — or the gradual deployment gateway policy — to selectively route traffic via the legacy network.

BGP route isolation — ASN community 32768

When BGP is used, routes learned from the legacy network can be propagated to other Cato sites — or kept isolated by tagging the prefixes with the Cato ASN and community 32768. The isolated-route option requires socket v15.

The interconnecting hub site

For phased migrations at scale, an interconnecting site is fundamental: one site holds both legacy CPE and a Cato socket, and carries traffic between the two networks while sites migrate wave by wave. A layer-3 switch (or firewall) at the hub makes the routing decision between networks.

Migrated ↔ legacy traffic via the hub Existing links
During phased migration the interconnect hub carries traffic between networks: Bristol (migrated) reaches Leeds (legacy MPLS) via the Cato backbone, the hub socket and the L3 switch that owns the routing decision.

Multi-regional hubs

Multi-region co-existence without asymmetric routing

With regional hubs on both networks, co-existence gets harder. Traffic initiated from an EMEA legacy site to an APAC Cato-connected site is forwarded via the APAC regional hub; the return traffic from APAC performs its own route lookup and comes back via the EMEA regional hub.

⚠ The failure mode: asymmetric routing

Return traffic follows a different regional path than the original flow. Stateful inspection on the hub firewalls sees only half the conversation — sessions fail and traffic drops during co-existence. Every multi-region design below exists to prevent this.

OptionApproachStrengthsWatch-outs
A Parallel connectivity, all sites. Deploy Cato in parallel across every site; temporary IPsec can precede sockets. Avoids cross-region latency and asymmetry; low-risk site-by-site cutovers with isolation; symmetry maintained throughout. Customer-side rework or site visits; may need additional PS hours. Preferred for fewer than 20 sites.
B Single hub + selective parallel. One primary hub is the temporary inter-network gateway; parallel connectivity only in selected smaller regions. Less complex than full parallel rollout; limits cross-region hairpinning; balances risk, cost and effort; can expand parallel coverage later. Hybrid option — suits estates where traffic patterns are uneven across regions.
C Phased regional rollout. Each region deploys independently; migrated sites temporarily hairpin via a single active hub, which shifts as regions complete. Isolates change to one region at a time; validation and rollback at regional boundaries. Deployment locked to regional phases; increased RTT through the intermediate transit region.
D NAT/PAT for symmetry. PAT preserves cross-regional routing symmetry between networks. No routing redesign; best where cross-region traffic volumes are low. Masks original source IPs — breaks protocols that embed them (e.g. LDAP, WMI); NAT config may be needed on customer infrastructure.
E Customer-managed PBR over GRE or IPsec tunnels. Granular traffic steering; symmetry maintained through policy routing. Highly complex; misconfiguration risks stability. Generally avoid unless strong in-house networking skills exist.
F VRF / eBGP interim connectivity. Separate legacy routing with VRFs, SD-WAN overlays or transit domains; regional eBGP between Cato and each hub. Symmetric routing preserved — cross-region routes carry a longer AS-PATH, so best-path selection anchors traffic to its regional hub. Simplified phased migration and rollback. Requires BGP-capable regional hubs and route-filtering discipline.
Symmetric flow — anchored to its regional hub Cross-region return — rejected by best-path Links & eBGP peerings
Option F: each region keeps its own eBGP handoff between the Cato hub site and the legacy hub. Cross-region routes carry a longer AS-PATH (an extra inter-hub hop), so BGP best-path selection anchors traffic — and its return — to the same regional hub.

Field example — Cato hub integration with FortiGate SD-WAN

Segmented legacy domains, interconnected with existing FortiGate regions via an eBGP handoff to each FortiGate hub. Path symmetry is maintained through shortest AS-PATH length, with routes dynamically propagated between the Cato and FortiGate networks. Spoke migration becomes mechanical: 1) disable the existing spoke, 2) turn up the new Cato spoke site, 3) routes propagate automatically, 4) rollback stays simple.

Interconnect & parallel-socket patterns

Routed ranges or BGP — the handoff catalogue

Two routing options carry traffic between networks at an interconnect or parallel-socket site: static routed ranges or dynamic propagation with BGP. These are the field-proven variants and their trade-offs:

PatternHow it worksStrengthsTrade-offs
Static hub — routed ranges Preconfigure routed ranges at the hub; prebuild Cato sites with dummy ranges (e.g. carrier-grade NAT space) and use the dummy objects in groups, firewall and network rules. To migrate: delete the routed range from the interconnect site, then update the dummy subnets at the new Cato site — rules and groups follow automatically. Simple to implement; predictable, lower risk; no reliance on BGP or customer filtering. Rollback = re-add the dummy range and the statics. Manual configuration at the interconnect for each migration.
BGP hub — L3 switch eBGP peer between Cato and the L3 device (works with firewalls or provider routers too); filter the default route and prefixes from sites with parallel connectivity. Migrate by disconnecting the legacy remote site's LAN — its subnets are withdrawn automatically and the new Cato site's prefixes propagate via BGP. Automatic route propagation; supports overlapping subnets and aggregate summaries; minimal config dependency at the interconnect site. Needs BGP capability and route-filtering discipline on the customer side.
Parallel socket — BGP handoff (LAN transit) eBGP peering between the socket and the customer firewall over a LAN transit VLAN; the firewall remains the default gateway. Phased subnet migration; dynamic route propagation; simplified rollback; Cato BGP routes can be propagated or isolated. Firewall stays in the traffic path during co-existence.
Parallel socket — L3 switch static The customer L3 switch makes the routing decision between legacy and migrated subnets; start with the SDP user range and system ranges via the socket, then move ranges as they migrate. Simple to configure; control stays on the switch; useful for validation; simplified rollback; keeps firewall throughput minimal. Route update required for every remote-site migration; increased rework; extended project timeline.
Parallel socket — FW static (LAN/DMZ transit) No L3 switch at site: sockets reach the internet via a DMZ transit off the existing firewall (which NATs socket WAN traffic to the PoP); the firewall remains DG and specific routes point via the Cato socket floating IP. Simplified route rollback; phased subnet migration. Route update per remote-site migration; additional firewall load from WAN encrypted traffic; rework required for WAN.
Proof at scale — ~300 sites

The static-hub routed-ranges strategy with dummy-range pre-staging is exactly how One global operator migrated nearly 300 sites: dummy CGN addressing prevented overlap with production subnets, and each migration was a delete-and-update of ranges rather than a rebuild of policy.

Socket as default gateway

When the socket owns the LAN from day one

Some designs put the Cato socket in as default gateway immediately — the legacy network becomes the exception that traffic is selectively routed to. Three options, each with a caveat worth knowing before the workshop:

Cato DC — BGP LAN handoff

  • Socket provides the default gateway; eBGP on a LAN handoff
  • HA failover fully supported
  • Dynamic route updates; routes propagated or isolated with the specific community
  • Allows use of the WAN firewall between local VLANs

Alt-WAN — BGP handoff

  • eBGP peer(s) via Alt-WAN steer legacy destinations over MPLS dynamically
  • Caveat: local routing ANY-ANY is required — otherwise Alt-WAN traffic first routes to the PoP, risking asymmetry
  • Caveat: Alt-WAN failure does not trigger socket HA failover — prefer the LAN BGP handoff where HA sockets are required
  • Routes propagated or isolated with community advertisement (v15)

Alt-WAN — gradual deployment GP

  • Longer-prefix-match RFC1918 destinations without specific routes forward via the Alt-WAN gateway
  • Reduces route updates; useful where a BGP handoff is unavailable or undesired
  • Caveat: only private (RFC1918) routes are supported — and they cannot be selectively enabled for specific private ranges
  • Same Alt-WAN HA limitation applies
