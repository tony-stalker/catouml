Identity Integration Design

Nearly every Cato deployment shares one dependency: the platform must know who is behind every flow before any identity-scoped policy can work. This page designs that dependency once — SCIM provisioning from Entra ID or Okta, SSO with MFA for the Client, and user awareness behind sockets — so every ZTNA rule, DLP scope and per-user event that follows lands on solid ground.

Business objective

Identity is the policy foundation

Every valuable thing the platform does downstream — a ZTNA rule scoped to contractors, a DLP policy scoped to finance, an event trail that names a person rather than an IP — depends on one upstream fact: Cato knows who the user is. Cato's model makes this worth designing properly: it maintains a single user identity, so once a user is added to a policy, “it is enforced whether the user is located behind a site or remotely”. One identity, one policy, both worlds.

Get identity wrong and the failure is quiet. Traffic from unrecognised users lands in the All Unidentified Users system group, group-scoped rules never match, and users fall through to whatever default rule sits at the bottom of the policy. That is the classic stalled rollout: the policy looks right, the users complain, and nobody can see why.

Access rules miss

ZTNA and firewall rules scoped to groups silently match nothing when the user behind the flow is unidentified — access is decided by the fallback rule, not the design.

Data scopes drift

DLP and CASB policies scoped to “finance” or “engineering” only bind if group membership is provisioned and current — stale sync means stale protection.

Events lose names

Investigations and audits need “Priya uploaded the file”, not “10.20.4.17 uploaded the file”. Attribution is only as good as awareness coverage.

Ask yourself

“If I pick any flow out of your firewall logs right now, can you tell me the person behind it — not the IP? And how many of your current rules are per-user exceptions rather than group-scoped policy?”

The design decisions

Four decisions, made once

Identity integration is four decisions. Each has a documented default; the point is to make all four deliberately, before any policy is written that depends on them.

DecisionOptions (documented)Design call
Directory & provisioning SCIM (Entra ID, Okta, OneLogin, DTS) · LDAP from on-prem AD · manual users SCIM from the corporate IdP — near real-time users and groups. Manual only for lab and break-glass accounts.
Authentication SSO via the IdP · Cato MFA (SMS / authenticator app) · username + password SSO with MFA enforced in the IdP flow — the IdP stays authoritative for credentials, MFA and conditional access.
User awareness behind sites Identity Agent (Cato Client) · AD Query (WMI on DCs) · Windows Event Forwarding Identity Agent wherever the Client can be deployed; AD Query or WEF to cover the clientless remainder.
Device posture Posture profiles evaluated alongside identity Design as identity's sibling — the same groups pair with posture profiles in ZTNA rules from day one.

1 · Directory & provisioning — SCIM

SCIM creates, updates and disables Cato users straight from the IdP, syncing changes in near real time — and, critically, it syncs user groups, the objects policy will reference. Enable it under Access → Directory Services (SCIM tab), then paste the Base URL and bearer token into the IdP's provisioning app.

  • Entra ID: “Cato Networks Provisioning” enterprise app, automatic mode, assign the specific users and groups to sync — Entra initiates sync roughly every 40 minutes. Nested groups are not supported: flatten them in the IdP.
  • Okta: enable Create / Update / Deactivate, then Push Groups to sync the groups policy will use.
  • Caveats: SCIM overrides existing LDAP groups with the same name; LDAP and SCIM provisioning cannot run simultaneously; unassigning a user in the IdP disables (not deletes) them in Cato.

Group strategy (recommended practice): build purpose-named, flat groups in the IdP for Cato to consume — ztna-contractors, dlp-finance — rather than reusing the org chart. Group names become part of your security policy; change-control them like firewall rules.

2 · Authentication — SSO + MFA

The authentication policy (Access → User Authentication) sets how SDP users authenticate account-wide — SSO, Cato MFA (SMS or an RFC-6238 authenticator app), or username and password — with per-user overrides for exceptions. With SSO, the PoP hands the user to the IdP, the user completes login including the IdP's MFA, and the PoP validates the IdP token before issuing its own Cato token — so conditional-access policy stays where it belongs, in the IdP.

  • Sixteen IdPs are supported for user SSO; Entra ID (Azure) and Okta cover all six authentication scenarios, including headless Linux.
  • Token validity: duration-based (silent renewal) or “Always Prompt”. Trusted devices skip MFA for up to 30 days — a geolocation change forces re-authentication.
  • Users activated with registration codes cannot use SSO or MFA — reserve codes for lab devices.

Protocol note: the current KB states Cato user SSO integrates with OIDC-compatible IdPs. If the estate is SAML-federated today, that is not a blocker — you integrate at the IdP (Entra ID, Okta…), which fronts Cato over OIDC — but surface it in design review rather than discovering it in week three.

3 · User awareness behind sockets

In the office there is no per-user tunnel — the socket carries everyone — so the PoP must learn who sits behind each IP. The documented methods, all configured under Access → User Awareness:

  • Identity Agent — the Cato Client runs in the background and reports the device's logged-in identity to the PoP about every 30 seconds, with IP changes reported immediately. With Windows Client v5.18+ / macOS v5.11+ this needs only a one-time authentication and no ZTNA licence. Best fidelity; first choice for managed fleets.
  • AD Query — Cato queries domain controllers over WMI for login events (dedicated AD account in Distributed COM Users + Event Log Readers; Directory Services configured first). Covers clientless estates — thin clients, machines the Client can't reach.
  • Windows Event Forwarding — DCs forward login events to a collector server that Cato queries; one collection point instead of WMI to every DC. Fits multi-DC estates and locked-down server policies.

Mixed estates mix methods. Design for coverage: every office subnet should have an awareness path, or its users will surface as unidentified.

4 · Device posture — identity's sibling

Access decisions pair who (identity and group) with the state of the device (posture profile: EPP/EDR running, disk encryption, OS version), evaluated continuously. Posture is not this page's story — see De-risking 3rd-Party Access for the full treatment — but it belongs in the same design session.

Define posture profiles under Access → Device Posture at design time, named with the same discipline as your groups, so ZTNA rules can pair group + posture from day one. Retrofitting posture into per-user exceptions later is the same anti-pattern as per-user identity rules.

How it fits together

One identity plane, every enforcement point

The four decisions interlock: SCIM builds the user and group objects in the Cato Management Application; SSO with MFA authenticates the remote user's Client; the Identity Agent (or AD Query) attributes office traffic behind the socket; and the PoP enforces the same group-scoped policy on both. The amber lane is what this design exists to prevent — a device with no awareness path, matching only All Unidentified Users and the fallback rules.

Identified & provisioned — group-scoped policy matches Unidentified — only fallback rules apply
Thin grey lines are the control plane: SCIM-provisioned groups become policy objects, and users authenticate to the IdP with SSO and MFA. Every data lane ends at the same identity-scoped policy — the amber lane is the coverage gap this design closes.

Sequencing

Identity before policy — the order that works

The dependency chain is strict: policy references groups; groups come from provisioning; users must authenticate and be recognised before any of it matters. Stage the work in that order, with exit criteria between stages.

StageDoExit criteria before the next stage
1 Provision. Enable SCIM, connect the IdP app, assign a pilot set of users and the purpose-built groups. Users and groups appear under Access → Users / User Groups with the IdP shown as the directory source.
2 Verify groups. Compare member counts against the IdP; flatten any nested groups at source (nesting is not supported over SCIM). Group membership matches the IdP and updates when you move a test user.
3 Authenticate. Set the authentication policy to SSO + MFA; pilot users connect with the Client end to end. Pilot users connect via SSO, MFA prompted in the IdP flow, sessions visible per user.
4 Attribute. Turn on the Identity Agent and/or AD Query so office traffic behind sockets carries a name. Office events in Monitor → Events show usernames, not bare IPs; unidentified traffic is trending down.
5 Only now: policy. Write (or migrate) ZTNA, firewall and DLP rules that reference the synced groups — never individual users. Rules match in monitoring; All Unidentified Users hit counts near zero.
The pitfall — migrating policy before identity is stable

Rules imported against groups that are empty, half-synced or misnamed match nothing — silently. Users fall through to defaults, the project team responds with per-user emergency exceptions, and those exceptions never get removed. Never translate a legacy per-user rulebase one-to-one: map it to groups first, prove the groups populate, then migrate. And treat the IdP group model as part of the security policy — rename a synced group and every rule that referenced it is orphaned.
