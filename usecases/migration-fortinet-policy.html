Fortinet Policy Migration

The network cutover from FortiGate to Cato is a solved routing problem — the policy estate is where migrations stall. This page works domain by domain through the FortiOS policy stack — firewall policies with their UTM profile pairs, webfilter profiles, inline CASB and DLP — showing what each construct becomes in the Cato Management Application, the semantic traps in between, and the four-phase process that rationalises the rulebase on hit counts before a single rule is translated.

Business objective

Why policy translation is the long pole

Sockets ship, BGP converges, sites cut over — and then the project meets a decade of accumulated FortiOS policy: per-VDOM policy tables, near-identical UTM profile variants, thousands of address and service objects, and rules nobody will own or delete. Translating that estate badly does more damage than translating it slowly: lift-and-shift faithfully reproduces every dead rule, every any-any exception and every profile carve-out in a brand new platform, and the customer inherits their old technical debt with a new logo on it.

The Cato Professional Services approach is a four-phase pipeline, applied per policy domain: EXPORT the configuration and its usage evidence → REVIEW & MAP — rationalise on hit counts, discard dead weight, then map surviving intent to Cato constructs → DEPLOY monitor-first, so every translated rule earns evidence before it can hurt anyone → OPTIMISE against events and hit counts until the policy is demonstrably right. Never lift-and-shift: the clean-up is not a nice-to-have, it is what makes the mapping tractable.

Ask yourself

“How many of your FortiOS rules have zero hits in the last 90 days — and who in your organisation is empowered to decide those rules don't migrate?”

Domain 1 — FW rules

FortiOS firewall policies & UTM profile pairs

The construct. FortiOS firewall policies are ordered, interface-pair rules in per-VDOM policy tables: source/destination interfaces (zones), address objects (firewall address / addrgrp), services, schedules and NAT settings — with UTM delivered as profile pairs attached per policy: webfilter, application control, IPS, anti-virus, DNS filter and an SSL/SSH inspection profile. Inspection happens only where a profile is attached. Central estates manage all of this as FortiManager policy packages per ADOM.

Export. Every FortiGate produces a flat-text CLI configuration — easy to parse, and therefore easy to script against: 

# Per device — full plain-text CLI configuration
execute backup full-config tftp <filename> <server>

# Central estates: export device configs and policy packages per ADOM from FortiManager.
# Usage evidence: policy hit-count / last-used columns, plus FortiAnalyzer / FortiView logs.

Site-to-site / overlay-zone policies WAN Firewall rules Default-block allowlist — matches the FortiOS implicit-deny model
Inter-VLAN / local east-west policies Socket LAN Firewall policy Enforced locally on the socket; or retain the FortiGate as an interim LAN firewall
firewall address / addrgrp Hosts, IP ranges, subnets and groups Global objects — dedupe before import; creation is scriptable via the GraphQL API or Terraform provider
service custom / service groups Custom services Protocol/port definitions translate almost mechanically
ISDB internet-service objects Applications and app categories Re-select by application intent — the catalogues do not share identifiers
Application control sensors App-aware Internet / WAN Firewall rules Application awareness is native to the rule, not a bolt-on profile
IPS sensor + AV profile pairs Cato IPS and Next-Gen Anti-Malware Account-level layers of the single-pass engine — per-policy carve-outs become explicit exceptions
Schedules Rule time constraints Re-model recurring windows per rule
Per-policy NAT / central NAT, VIPs PoP egress NAT; remote port forwarding / static IP for inbound Inbound published services move on their own change windows
VDOM scoping Policy sections and site/segment-scoped rules in one rulebase Separate Cato accounts only where hard tenancy is required

Semantic gotchas. One FortiOS policy table becomes layers: a single policy with its UTM profile pairs decomposes into a firewall rule, account-level IPS/NGAM settings and a TLS inspection rule — agree the layer mapping before translating rule one. Interface-pair semantics have no equivalent; recover the intent (which segment reaches what) rather than transliterating srcintf/dstintf. And traffic that escaped inspection on FortiOS because no profile was attached will be inspected by Cato's single-pass engine — expect new detections in the monitor phase and triage them before enabling block. Flow-mode versus proxy-mode inspection differences belong on the discovery list, validated in the PoV rather than assumed.

Strategy. Parse the flat-text config; retire zero-hit rules and collapse duplicate objects against the hit-count evidence; push the cleaned object model through the API or Terraform first, then build the rule layers on top of it; deploy new WAN Firewall and Internet Firewall rules monitor-first and let events prove them before enforcement.

Domain 2 — SWG policy

Webfilter profiles & FortiGuard categories

The construct. Web filtering lives in webfilter profiles attached per policy: FortiGuard category actions (allow, monitor, warn, authenticate, block), static URL filter lists (wildcard and regex), rating overrides and options such as safe search and browsing quotas — usually accumulated as several near-identical profile variants, one per user population. A DNS filter profile often sits alongside, enforcing similar categories at the resolver.

Export. The webfilter profile, webfilter urlfilter and DNS-filter sections of the config backup, plus the FortiManager object database; pull web-usage reporting from FortiAnalyzer as the evidence base for what each category action actually affects.

FortiOS constructCato constructNotes
FortiGuard web categories Cato URL categories in Internet Firewall rules Taxonomies differ in names and granularity — map by intent, review every blocked category individually
Category action: block Block (with block page) Deploy contentious categories as monitor first
Category action: warn / continue Prompt action User coaching — proceed after acknowledging the warning
Category action: authenticate Identity-scoped rules No direct equivalent needed — every Cato rule is identity-aware natively
Static URL filters & rating overrides Custom categories Consolidate allow/block lists and re-categorisations into custom categories referenced by rules
Per-population profile variants User/group-scoped Internet Firewall rules Depends on IdP/SCIM group provisioning — sequence it first
Browsing quotas Re-model as Prompt or time constraints No per-user browsing-quota construct — decide the intent with the customer
DNS filter profile Inline category enforcement at the PoP One policy governs the flow — no separate resolver-layer policy to keep in sync

Semantic gotchas. The taxonomy mismatch is the headline risk: a FortiGuard category and a similarly-named Cato category are not guaranteed to cover the same sites, so a mechanical name-for-name mapping silently changes what is blocked. Fidelity is the second: without TLS inspection both platforms act on domain/SNI only, but full-path URL matching and content-level decisions need decryption — stage TLS inspection before judging SWG parity. Finally, profile sprawl should die here: collapse the near-identical variants into a small number of group-scoped rule tiers instead of recreating each one.

Strategy. Build a category translation sheet mapping each FortiGuard category-action to a Cato category and action by intent; consolidate static lists into custom categories; deploy the blocked set in monitor mode and compare events against FortiAnalyzer web reports; flip to block per category once the evidence matches expectations, using Prompt where the customer used warn pages.

Domain 3 — CASB policy

Inline CASB profiles

The construct. Recent FortiOS releases add inline CASB profiles — SaaS application access and tenant controls driven by FortiGuard SaaS signatures, attached per policy like every other UTM profile. In most estates the capability arrived inside a FortiGuard bundle and was never operationalised: expect a handful of configured controls at most. This is an intent-mapping exercise, not an object migration.

Export. The CASB and SaaS-application sections of the config backup — often near-empty — plus FortiView SaaS usage reporting, which is the more valuable artefact: it shows what the organisation actually uses, sanctioned or not.

FortiOS constructCato constructNotes
Inline CASB profile app controls Cato CASB application rules Granular activity control — upload, download, share, log-in — per app and user group
Tenant restriction Cato tenant control rules Corporate-tenant-only access to sanctioned suites
FortiGuard SaaS signatures Cato application catalogue with risk attributes Different catalogues — re-select applications by name and intent
FortiView SaaS usage reporting Cloud Apps Dashboard / App Analytics Shadow-IT discovery with app risk scoring, on by default

Semantic gotchas. Activity-level control on either platform requires TLS inspection — a CASB "migration" judged before decryption is staged will look falsely thin. And because the FortiOS side is usually barely configured, the real risk is anchoring scope to the old config rather than to discovered usage: migrate what the organisation needs, not the two profiles it happened to have.

Strategy. Discovery first — let the Cloud Apps Dashboard build the real SaaS usage picture during the monitor phase, agree the sanctioned-app list with the customer, rebuild the few genuine inline controls as Cato CASB rules, then tighten from monitor to enforcement. API-based SaaS scanning (FortiCASB-style) is a separate design conversation to scope with Cato Professional Services.

Domain 4 — DLP policy

DLP profiles & sensors

The construct. FortiOS data-loss prevention is built from DLP profiles and sensors: file-type and file-size filters, dictionaries and regex data patterns, optional document fingerprints, with log-only or block actions — attached per policy like the other UTM profiles. Deeper content inspection has historically required proxy-mode processing with its performance cost, so many estates run DLP log-only on a few policies, or not at all.

Export. The dlp profile / dlp sensor and dictionary sections of the config backup. The custom regex patterns and keyword dictionaries are the only assets worth carrying forward — capture them verbatim.

FortiOS constructCato constructNotes
DLP sensors / data patterns Cato DLP profiles Prefer Cato's predefined data types (PCI, PII, credentials, source code) over ported regexes
Custom dictionaries & regexes Custom data types Regex dialects differ — re-test every pattern against sample data
File-type / file-size filters File attribute conditions in DLP rules Combine with data types for precision
Per-policy profile attachment Global DLP policy scoped by user, app and direction Upload/download rules per application and group, enforced in the cloud
Document fingerprinting Validate the requirement with Cato PS Map the intent in discovery rather than assuming parity

Semantic gotchas. This is a rebuild, not a conversion — there is no importer, and that is the right outcome: the per-policy, proxy-mode-constrained FortiOS model does not map onto a cloud DLP engine that inspects everything the TLS policy decrypts. TLS inspection is the hard prerequisite; regex behaviour differs between engines; and because install-base usage is typically thin, this migration is in practice an upgrade — from log-only sensors to an actual data-protection programme.

Strategy. Inventory the sensors actually referenced by policies; recreate custom patterns as custom data types and swap standard patterns for Cato's predefined types; run the whole policy in monitor with the Data Protection Dashboard as the evidence loop; tune false positives with the data owners; then enforce rule by rule.

Several Before rows have no After row — deliberately. The corp-ips and av-default profiles on policy 12 are no longer per-rule attachments: they become the account-level IPS and Next-Gen Anti-Malware layers of the single-pass engine, applied to everything it sees. There is no "Corp-Internet-Access" allow rule either — the Internet Firewall's default-allow posture does that job, so what survives of policy 12 is its webfilter intent, as rule 3. Policy 3 exits the pipeline at REVIEW & MAP on its hit-count evidence, and policy 0 — the implicit deny — is a posture decision, not a rule: the WAN Firewall is implicit-block already, and if the customer wants allowlist egress it becomes an explicit final block rule.

What changed in translation

  • Profile pairs became layers. The webfilter + IPS + AV pair attached to policy 12 decomposed into an Internet Firewall category rule plus account-level IPS and NGAM — separate global policy layers of the single-pass engine, with the old per-policy carve-outs re-expressed as explicit, auditable exceptions.
  • Interface pairs became scopes. internal → wan1 has no equivalent construct — the intent (which segment reaches the internet, under what constraints) is recovered as source scopes in one global rulebase, as the guest rules show.
  • Categories were mapped by intent, not name. The FortiGuard categories behind default-webfilter were re-selected individually against the Cato taxonomy — similarly-named categories do not cover the same sites — and deployed monitor-first before enforcement.
  • Hit counts decided what never made the journey. The zero-hit legacy rule was discarded at review rather than translated — the discard pile, and the posture decision that replaces the implicit deny, are outputs of the migration as much as the rules that survive.
Cross-domain practices

Rules that hold across all four domains

🧭

Respect the default postures

Cato's WAN Firewall is an allowlist (implicit block); the Internet Firewall is a blocklist (implicit allow). FortiOS is implicit-deny everywhere — so egress posture is a decision, not a translation. If the customer wants allowlist egress, add an explicit final block rule; don't discover the difference in production.

🔐

Stage TLS inspection early

Without decryption, SWG acts on domain/SNI only and CASB activity control and DLP content inspection cannot see payloads. Phased TLS inspection — by category and pilot group, certificates deployed in advance — is the fidelity prerequisite for three of the four domains.

🟠

Monitor, then block

Every translated rule, IPS/NGAM protection, CASB control and DLP rule starts in monitor. Enforcement is earned with event evidence, per rule — the pipeline's amber-to-green transition, never a big-bang flip.

🪪

Sequence identity first

Group-scoped rules are only as good as the groups. Provision IdP/SCIM sync before policy deployment, and map FSSO/LDAP group references in FortiOS policies to the synced directory groups — name drift here breaks rules silently.

🧱

Objects before rules

Translate and dedupe the address/service object model first — scriptable from the flat-text config via the GraphQL API or Terraform provider — then build rules on the clean object base. Rules referencing ad-hoc objects recreate the sprawl you just removed.

📊

Validate with events & hit counts

The evidence loop closes in Monitor → Events: compare each translated rule's hits against its FortiOS ancestor, investigate the deltas, and treat zero-hit rules after a full business cycle as candidates to delete — on the new platform as ruthlessly as on the old.

The policy pipeline

EXPORT → REVIEW & MAP → DEPLOY → OPTIMISE

The same pipeline runs for each domain, offset in time — firewall first, SWG behind it, CASB and DLP once TLS inspection and discovery data are in place. Dead rules leave the pipeline at review and are never translated; everything that survives deploys in monitor and is promoted to enforcement on evidence.

Monitor first — log, don't block Enforce — promoted on evidence Discarded — dead rules never migrate
The four FortiOS policy domains are exported once, rationalised against hit-count evidence — dead rules and any-any debt exit the pipeline here — and only the surviving intent is mapped to CMA constructs. Every mapped rule deploys in monitor (amber) and is promoted to enforcement (green) when events and hit counts prove it behaves as intended.
