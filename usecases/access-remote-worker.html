Remote Worker Experience: Per-User, Per-App Assurance

Estate dashboards average away the one person who is actually having a bad day. This use case follows a single remote worker — from onboarding, through the always-on Cato Client and nearest-PoP egress, to the morning they say “it is slow” — and shows how per-user, per-app Experience Monitoring turns that sentence into an evidence-backed fix: last mile, backbone or the app itself, named in one screen.

On this page
Business objective

Why this matters

Two other pages in this library carry the wide-angle versions of this story: Securing the Hybrid Workforce owns the access policy — one consistent, secure front door for every user — and Proactive Digital Experience Monitoring owns the organisation-wide view: sites, probes, path analysis and anomaly detection across the estate. This page deliberately zooms all the way in: one named worker, one laptop, one home connection — because that is the granularity at which remote work is actually experienced, and at which support tickets are actually raised.

The objective: for any individual remote worker, at any moment, be able to say what their experience of each application is, and — when it degrades — which segment of their path is responsible. Not “the network looks fine”; a per-user, per-app answer with numbers behind it.

The problem

One user, zero evidence

Estate-level monitoring is built to spot patterns — a degraded site, a failing circuit, a SaaS outage. A single remote worker generates no pattern. Their home Wi-Fi, their ISP and their laptop are invisible to corporate tooling, so when they report a problem the investigation starts from nothing.

The anecdote trap

A remote worker's ticket is one person's word against a green dashboard. With no telemetry from the home, the helpdesk falls back on ritual: reboot, speed test, “is anyone else seeing this?” — and because nobody else is, the ticket closes as no fault found while the worker's day stays slow.

The per-app blind spot

“Slow” is rarely uniform. One SaaS app can crawl while everything else is fine — an app-side problem — or every app can degrade together, which points at the device or the last mile. Without per-app scores for that one user, the two cases are indistinguishable, and each gets escalated to the wrong team.

Ask yourself

“Pick one home worker who called the helpdesk last week. Could you see what their Microsoft 365 experience was that morning — as a number, per application — or only what the service health page said for everyone?”

The worker's journey

Brandon, from day one to steady state

Meet Brandon — a fictional remote worker on the US west coast, hired by a UK-headquartered organisation that never intends to put a VPN concentrator between him and his work. His journey is four decisions, each made once in policy and enforced automatically ever after:

StageWhat happens for BrandonWhere it lives in CMA
Day 1 — onboarding The Cato Client is deployed with his laptop build; he signs in with corporate SSO and his provisioned identity (see Identity Integration Design). Access → Users
First connect The Client tunnels automatically to the nearest available PoP — for Brandon, Santa Clara — and the Always-On policy keeps it that way. Access → Client Connectivity Policy
Egress choices By default everything rides the tunnel for full inspection; the split tunnel rule base defines any deliberate exceptions. Access → Split Tunnel Policy
Continuous posture Device checks (anti-malware, disk encryption, patch level) gate his access — evaluated continuously, not just at sign-in. Access → Device Posture
Steady state Every sanctioned application he touches is scored Good / Fair / Poor, per app, continuously — before he ever complains. Home → Experience Monitoring

The three policies behind the quiet steady state

Always-On

An ordered rule base ties users and groups to mandatory Cato connectivity, so traffic is always inspected. Escape hatches are deliberate, not accidental: admin-issued one-time bypass codes (valid up to 15 minutes), user-initiated bypass with a recorded reason (Windows Client v5.9+, macOS v5.5+, iOS v5.6+), and a recovery mode that decides whether internet access continues if the Cato Cloud is unreachable. Not supported on Linux Clients.

Split tunnel & egress

Default: all traffic through the tunnel to the nearest PoP for full inspection. The Split Tunnel Policy is an ordered rule base with two deliberate alternatives — exclude mode (everything to Cato except named exceptions) and include mode (only named traffic to Cato) — scoped by user or group, OS, country and managed/unmanaged network, matching IP ranges, domains, applications and ports, with local DNS resolution for chosen domains.

Connectivity + posture

The Client Connectivity Policy checks the device before it joins: posture profiles bundle device checks with AND logic — anti-malware present, disk encryption on, patches current (Windows and macOS) — alongside platform, country and authentication confidence level. A device that stops complying drops to secured-internet-only access or is blocked, per rule.

The point for the room: Brandon never files a “please set up my VPN” ticket, never picks a gateway from a dropdown, and never works uninspected. The same policies also generate the telemetry the next section runs on — the Client and the PoP see every packet of his working day.

The Cato solution

When Brandon says “it is slow”

Six months in, Brandon messages the helpdesk: “Office 365 has been slow all morning.” Because his Client, his nearest PoP and the backbone between them are one platform, every segment of his path is already instrumented — the answer is a look-up, not an investigation. The diagram shows where the measurements come from:

Encrypted tunnel — Client to nearest PoP, then backbone to each app
One platform observes every hop of Brandon's path — so a Fair score can be pinned to a segment. Loss at ② with a clean ③ and ④ is his ISP; a slow ④ with ①–③ green is the application's side of the fence.

In the CMA that logic lands on one page: Brandon's user drill-down in Experience Monitoring. The summary strip identifies the session — user, site (Remote User), last connected PoP and ISP, plus his device's OS, Client version and internal/external IPs. The Select Application widget ranks his top applications by usage with an average experience score for each, and selecting one reveals its per-app measurements: Time to First Byte, TCP connect, TLS connect, HTTP latency and HTTP/S errors. For Zoom, Teams and Webex, a meetings widget breaks each call down by audio, video and screen sharing — packet loss, jitter, latency and bitrate.

When the complaint is “everything is slow” rather than one app, the Connection Details view decomposes his path into scored nodes — client device, Wi-Fi, LAN gateway, last mile (tunnel) and application — each independently rated against published thresholds: CPU Good below 70% utilisation, Wi-Fi Good at −67 dBm or stronger, LAN gateway and tunnel Good below 1% loss and 10 ms. The first non-green node names the fault domain; a tunnel-age widget adds context without skewing the score.

🧑‍💻

Per-user, per-app truth

Every sanctioned app Brandon uses carries its own Good / Fair / Poor score — so “slow” resolves to which app, how much, since when before anyone remotes into his laptop.

📍

Segment isolation in one screen

Device, Wi-Fi, LAN gateway, last-mile tunnel and application are scored as separate nodes — last mile vs backbone vs app-side is a glance, not a war room.

🔁

Telemetry as a by-product

No extra agent, no synthetic-only view: the always-on Client and the PoP that carry Brandon's traffic are the sensors, so coverage is exactly his working day.

🧾

Evidence for every audience

TTFB and flow counts for the app team, tunnel loss for the ISP call, posture and block events for security — one drill-down feeds them all.
