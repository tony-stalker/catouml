iboss Policy Migration

The steering swap gets the headlines; policy translation is where an iboss replacement actually stalls. This page works through the four iboss policy domains — cloud firewall rules, web policy with its education-grade content controls, CASB and DLP — and maps each one to its Cato CMA construct: the export to take, the semantic traps, and a monitor-first deployment path aligned to per-OU cohort waves with PAC kept hosted as the rollback rail.

Business objective

Policy translation is the long pole

Read this first — research, not a delivery-verified runbook

This page is authored from sourced vendor and Cato documentation research, not from a completed customer migration. Review the mappings and the deployment sequence with Cato Professional Services before using them in a customer engagement.

Sockets and Clients move traffic in weeks; the years of accumulated policy are what keep a legacy tenant licensed for months. An iboss estate carries per-OU web policy tiers, allow/block lists, SSL decryption bypasses, CASB and DLP rules — much of it built up exception by exception since the appliance-filter days, with nobody left who remembers why half of it exists. The instinct to "just recreate everything in Cato" is how migrations inherit a decade of technical debt on day one.

The Cato PS approach is a four-phase pipeline: EXPORT the policy and objects per domain → REVIEW & MAP — rationalise on hit counts, discard dead and duplicate rules, then map the survivors to CMA constructs → DEPLOY monitor-first — every translated rule lands tracking events, not blocking → OPTIMISE from live telemetry before enforcement. Never lift-and-shift: a rule that hasn't matched traffic in a year is documentation, not policy, and it dies in the review phase — not in the CMA.

Throughout this page the deployment unit is the cohort — an OU, year group or school — matching the per-OU policy tiers iboss estates already run. Policy waves ride the same cohort waves as steering, and because the iboss PAC stays hosted and unchanged for the whole programme, any cohort can be re-pointed to the old world with a single GPO/MDM change while the policy question is worked.

Domain 1 — FW rules

iboss cloud firewall → Cato Internet firewall

iboss bundles FWaaS at every licence tier, but in a proxy-heritage estate the firewall layer is usually modest: port/protocol controls for the GRE/IPsec-tunnelled guest, BYOD and IoT traffic that can't run an agent, plus a handful of non-web protocol rules riding alongside the web policy. Export: pull the rule set from the iboss admin console as part of the Phase-0 policy export (rules, source groups, destination and port/protocol objects). Where console exports are thin, transcribing a modest rule base by hand is legitimate — validate the count with the customer's administrator so nothing is missed.

iboss constructCato constructMapping note
Cloud firewall rule (source group → destination, port/protocol, action)Internet firewall rule — user group / site source, destination object, service, Allow/Block + trackOne global ordered policy replaces per-gateway rule sets
Port/protocol objectsCato services (standard or custom)Prefer application-aware rules over raw ports where an app object exists
Destination host/IP listsIP range objects / custom appsCreate objects before rules — object import precedes policy creation
Per-gateway or per-location variantsSite- or group-scoped rules in the single global policyDuplicated per-location rules consolidate to one rule with multiple sources
Rules reaching private/internal appsWAN firewall + Private Access policy — not the Internet firewallDifferent policy, different default posture (see cross-domain practices)

Semantic gotchas. Cato runs two firewalls with opposite defaults: the Internet firewall is a blocklist with a final implicit allow; the WAN firewall is an allowlist with a final implicit deny. An iboss egress rule base built as default-deny cannot be transplanted rule-for-rule into the Internet firewall without deciding the target posture explicitly — either recreate the deny stance as a final block-all rule with allow rules above it, or (usually better) adopt the Cato best-practice block set and keep the implicit allow. Zone-ish per-gateway context also disappears: reverse-engineer what each gateway's rules were really scoping before you consolidate them.

Ask yourself

"When did anyone last review firewall rule hit counts on the iboss side — and how many of those rules would survive a zero-hit cull?" The answer sizes the REVIEW phase and usually shrinks the migration by a third before mapping starts.

Migration strategy. Rationalise first — dead, disabled and duplicate rules are discarded at review, not migrated. Map the survivors to application-aware Internet firewall rules, deploy the translated policy with every rule tracking events, and let a full cohort wave of telemetry prove each rule matches the traffic it was translated for before any block action goes live.

Domain 2 — SWG policy

Web policy: the deepest translation

This is the heart of an iboss estate and the deepest section of the workshop. The constructs: per-group web policies aligned to OUs — year groups and staff tiers in education — each with category allow/block decisions, URL allow/block lists, SafeSearch enforcement, YouTube restriction tiers, and an SSL decryption policy with its accumulated bypass lists. Export: categories, allow/block lists, per-group overrides and SSL decryption bypass lists from the iboss console — and harvest the PAC file exceptions alongside them. The PAC and bypass lists encode years of "this app breaks under decryption" tribal knowledge: they are the cheapest policy-mining asset in the whole project and the seed for the Cato TLS inspection bypass rules.

iboss constructCato constructMapping note
Web categories per group policyInternet firewall rules — URL categories, source = SCIM user groupTaxonomies differ: map category-by-category, never assume 1:1
URL allow/block listsCustom categories and rule exceptions in the Internet firewallRationalise on hits first — old exception lists are mostly dead entries
Per-OU policy tiers (year group / staff)Ordered rule blocks scoped to user groups in one global policyRule order replaces per-group policy objects — document the tier ladder
SafeSearch enforcementContent Restrictions — SafeSearchApplies across supported search engines; see the DoH caveat below
YouTube restriction tiersContent Restrictions — YouTube Moderate / StrictMap each iboss tier to Moderate or Strict explicitly per cohort
SSL decryption scope + bypass listsTLS Inspection policy + bypass rulesSeed bypasses from the harvested iboss/PAC lists, then tighten from events
iboss root MITM certificateCato root certificate, pushed via MDMBoth CAs side by side from day one; the iboss CA is removed last, at decommission

The CA swap is the fidelity prerequisite. Category matching without decryption is SNI-level; the education-grade controls and everything in the CASB and DLP domains below need TLS inspection to see payloads. Push the Cato root certificate to all endpoints alongside the existing iboss CA before any policy wave moves — Cato documents the certificate as mandatory for TLS inspection, downloadable from the CMA and deployable via MDM. Then stage inspection the documented way: prerequisites (certificates deployed, QUIC blocked, test groups defined) → initial policy with source/destination bypasses → phased enablement per cohort → ongoing tuning of bypass rules from certificate-error events.

The DoH caveat — be upfront about it

Cato documents that browsers using DNS-over-HTTPS bypass Content Restrictions. SafeSearch and YouTube enforcement therefore depend on controlling DoH — via browser management policy, and by blocking the DNS-over-HTTPS service in the Internet firewall per Cato best practice. Put the DoH control in the same change as the Content Restrictions rules, not after the first bypass incident.

Semantic gotchas. Category taxonomy mismatch is the big one — an iboss category and a similarly-named Cato category are not guaranteed to cover the same sites, so parity is proven with a test URL list per cohort, not asserted from names. For fine-grained Google-service controls in education rulesets, run a feature-parity check against the customer's actual iboss configuration rather than claiming equivalence. Decide the uncategorised-site stance deliberately (Cato best practice is to restrict uncategorised destinations). And remember education customers treat per-user reporting as statutory safeguarding evidence — user-attributed CMA events are part of policy parity, not an afterthought.

Ask yourself

"How many distinct web policy tiers do you actually run across your OUs — and which pairs differ by more than a handful of rules?" Tiers that differ cosmetically collapse into shared rule blocks during mapping; tiers that differ materially define the cohort wave plan.

Migration strategy. Build the baseline Internet firewall from the Cato best-practice block/prompt set, then layer the per-OU tiers as group-scoped rule blocks in tier order. Deploy each cohort's web policy in monitor alongside its steering wave, run the category-parity URL list and the SafeSearch/YouTube checks with safeguarding staff verifying, and only then flip block actions live. PAC rollback stays available for the whole exercise.

Domain 3 — CASB policy

iboss CASB → App Control & tenant restrictions

iboss provides inline CASB: shadow-IT visibility and cloud-app controls enforced in the proxy path — and, distinctively, many customers enforce SaaS tenant restrictions on the SaaS side, keyed to their iboss dedicated egress IPs. Export: the CASB rule set from the policy export, plus a dependency register of every tenant restriction, exam-platform allowlist and partner firewall rule keyed to the dedicated IPs — the register matters more than the rules.

iboss constructCato constructMapping note
Cloud app allow/block controlsApp Control (CASB) rules — app, activity, user group, actionCato adds granular activity control (e.g. upload vs view) — translate intent, then refine
Shadow-IT visibility / app discoveryApp Analytics and Cloud Apps Dashboard in the CMABaseline both catalogues in discovery — coverage differs in both directions
App risk ratingsCato app risk scores in App AnalyticsGartner flagged the small iboss SaaS risk catalogue — re-score, don't copy ratings
Tenant restriction via dedicated egress IPsAllocated IPs + NAT egress rules, plus in-policy tenant enforcementDual-list both vendors' IPs on the SaaS side before each wave moves

Semantic gotchas. The app catalogues will not line up — expect apps iboss names that Cato classifies differently, and vice versa; resolve mismatches during mapping, not during an incident. Activity-level CASB control requires TLS inspection on the relevant traffic, so the CASB wave inherits the SWG domain's inspection scope. And the dedicated-IP dependency is the classic "migration broke the exam platform" incident: the default Cato licence includes three allocated IPs, so size the egress-IP design early if the estate needs more or regional egress.

Migration strategy. Start from Cato's default recommended CASB policy rather than a blank page, translate the surviving iboss app rules into it, and run the whole domain in monitor. Review App Analytics against the customer's iboss shadow-IT expectations for a full cohort wave, tighten activity rules from what monitor mode reveals, then enforce per cohort — with the SaaS-side allowlists already dual-listed.

Domain 4 — DLP policy

iboss DLP patterns → Cato DLP profiles

iboss DLP inspects data in motion against patterns and dictionaries in the proxy path. Export: capture the rule intent, not the engine — for each iboss DLP rule, document the data types it protected (PII, safeguarding records, finance data), the scope (which groups, which apps and directions) and the action. Engines and match logic differ, so the patterns themselves do not port: rebuild against Cato's content profiles rather than exporting regexes.

iboss constructCato constructMapping note
DLP patterns / dictionariesDLP content profiles — 350+ predefined data types + custom typesRebuild by data-type intent; do not port regexes blindly
DLP rules (scope, direction, action)Data Control policy — app/user-group scope, upload/download, actionTranslate scope to SCIM groups; actions start as monitor
Per-user incident reportingData Protection Dashboard + user-attributed DLP events; SIEM exportDemo user-level evidence early — it is the safeguarding/compliance expectation

Semantic gotchas. A rebuilt profile has a different false-positive signature to the rule it replaces — the old rule's precision tells you nothing about the new one, which makes monitor-then-block mandatory here, not merely prudent. DLP content matching also depends entirely on TLS inspection scope: a destination bypassed for inspection is invisible to DLP, so review the bypass list from the SWG domain with DLP eyes before trusting coverage.

Migration strategy. Recreate each surviving rule's intent from content profiles, run the Data Control policy in monitor across a full cohort wave, review hits in the Data Protection Dashboard with the data owners, tune profiles until precision is proven, then flip rules to block one at a time. Archive the iboss DLP incident history before the legacy subscription terminates — in education it is statutory evidence.

What changed in translation

  • OU scoping → SCIM group scoping. The per-OU policy objects become group-scoped rules in one ordered global policy — Students rules sit above Staff in the tier ladder, and the groups must exist via SCIM before any of these rules can be built.
  • SafeSearch and YouTube move to Content Restrictions. Same intent, new dependency: enforcement holds only if DoH is controlled — browser management policy plus the DNS-over-HTTPS block in the Internet firewall, deployed with the rule, not after the first bypass.
  • Schedules carry over as rule time conditions — verify, don't assume. The school-hours window translates to a time condition on the CMA rule, but this page's mappings are research-based and its gotcha sections don't cover schedule semantics: treat timezone handling and term-date exceptions as exactly the kind of behaviour the monitor wave exists to prove before the block goes live.
  • Bypass lists → TLS inspection bypasses and rule exceptions. The exam-platform PAC and SSL entries seed a TLS Inspection bypass rule (plus an allow above any category block that would catch it) — remembering that a bypassed destination is invisible to CASB and DLP, so keep the bypass as narrow as the harvested list allows.
Cross-domain practices

Rules that hold across all four domains

🧭

Know the two default postures

Cato's WAN firewall is an allowlist (implicit deny); the Internet firewall is a blocklist (implicit allow). Every translated rule must land in the right policy with the posture decision made explicitly — transplanted defaults are silent policy bugs.

🔐

Staged TLS inspection first

Inspection fidelity is the prerequisite for SWG category accuracy, CASB activity control and DLP content matching. Certificates → initial bypasses → phased enablement → tune. No decryption, no policy parity.

👁️

Monitor, then block

Every translated rule deploys tracking events with no block action. A full cohort wave of telemetry proves precision; enforcement is flipped rule-by-rule on evidence, never wholesale on a date.

🪪

Identity before policy

Per-OU rules need the groups to exist first: SCIM provisioning (Entra ID, Okta, OneLogin) feeds the user groups every domain scopes to. Google Workspace schools need the identity design agreed in discovery — it gates the whole policy build.

📊

Validate with events and hit counts

Hit counts rationalise the old estate and validate the new one: zero-hit legacy rules die at review, and translated rules earn enforcement by matching the traffic they were built for in CMA events.

⟲

Cohort waves, PAC-safe

Policy moves with steering, one OU / year group / school at a time, each device wholly on one vendor. The iboss PAC stays hosted and frozen throughout — rollback for any cohort is one GPO/MDM change.

The policy pipeline

Export → rationalise → map → monitor → enforce

One pipeline, run per domain, per cohort wave. Dead rules are discarded at rationalisation so the CMA starts clean; everything that survives deploys in monitor and earns enforcement from telemetry.
