Check Point Policy Migration

The network cutover from Quantum gateways to Cato is a solved routing problem — the policy estate is where migrations stall. This page works domain by domain through the SmartConsole policy stack — Access Control layers and the NAT rulebase, the URL Filtering and Application Control blades with their HTTPS Inspection dependency, inline CASB intent and the DLP blade — showing what each construct becomes in the Cato Management Application, the semantic traps in between (layer semantics first among them), and the four-phase process that rationalises the rulebase on hit counts before a single rule is translated.

Business objective

Why policy translation is the long pole

Sockets ship, BGP converges, sites cut over — and then the project meets a decade of accumulated SmartConsole policy: Ordered Layers chained so that every layer must pass, Inline Layers nesting parent and sub-rules, layers shared across policy packages so different teams could edit them, a NAT rulebase living entirely apart from access policy, thousands of objects and per-gateway blade carve-outs. Translating that estate badly does more damage than translating it slowly: lift-and-shift faithfully reproduces every dead rule, every any-any exception and every layer of delegation debt in a brand new platform, and the customer inherits their old technical debt with a new logo on it.

The Cato Professional Services approach is a four-phase pipeline, applied per policy domain: EXPORT the configuration and its usage evidence → REVIEW & MAP — rationalise on hit counts, discard dead weight, then map surviving intent to Cato constructs → DEPLOY monitor-first, so every translated rule earns evidence before it can hurt anyone → OPTIMISE against events and hit counts until the policy is demonstrably right. Never lift-and-shift: the clean-up is not a nice-to-have, it is what makes the mapping tractable.

Ask yourself

“How many of your access-control rules have zero hits in the last 90 days — and which of your layers exist because traffic needs a boundary, rather than because a past administrator needed delegation?”

Domain 1 — FW rules

SmartConsole Access Control layers & NAT

The construct. Check Point access policy is a hierarchy, not a list. A policy package holds one or more Ordered Layers — a packet must pass every layer, so an Accept in layer one only stands if no later layer drops the flow — and each layer carries its own implicit cleanup action. Inline Layers nest a sub-rulebase under a parent rule: match the parent but no sub-rule and the inline layer's cleanup drops you, even where a flat rulebase would have kept matching. Layers can be shared across packages and delegated to different administrator teams, objects live in a central database, and NAT is a separate rulebase — automatic (object-driven) and manual rules evaluated first-match, independent of access policy. Multi-Domain estates repeat all of this per domain.

Export. The Management API pulls each layer with hit counts and resolved objects — the raw material for rationalisation: 

# Per layer — rulebase with hit counts, objects resolved via a dictionary
mgmt_cli show access-rulebase name "Corp-Policy Network" use-object-dictionary true \
  show-hits true limit 500 offset 0 --format json

# limit caps at 500 — paginate with offset. One layer per call: inline layers are
# queried separately. On Multi-Domain (MDS), repeat per domain.
# Full offline export: CheckPointSW/ExportImportPolicyPackage (R80.10+).
# Also capture: NAT rulebase, Threat Prevention profiles, HTTPS Inspection policy,
# VPN communities and Identity Awareness sources.

Semantic gotchas. Layer semantics are the number-one gotcha of the whole migration. An inline layer's implicit cleanup differs from a flat rulebase: a packet that matches the parent rule but no sub-rule is dropped there and then, and an Accept in an ordered layer merely continues matching into the next layer — so a mechanical rule-for-rule copy silently changes what is allowed. Translate the effective policy — what actually passes every layer — never the layer structure, and budget conversion effort by the number of layers and inter-layer dependencies, not the raw rule count. NAT is its own rulebase evaluated apart from access policy: inventory automatic and manual rules separately, and treat every DNAT-published service as its own change window with a return-path check. Finally, layers are often delegation artefacts — in CMA that intent becomes rule sections, scoped admin roles and the audit trail, not policy structure.

Strategy. Export every layer with hit counts and the object dictionary; strip zero-hit, disabled and shadowed rules before any mapping — zero-hit extraction is scriptable against the API export; model the effective policy across layers before touching CMA; push the deduped object model through the GraphQL API or Terraform first; then deploy WAN and Internet Firewall rules monitor-first and let events prove them before enforcement.

Domain 2 — SWG policy

URL Filtering & Application Control blades

The construct. On R8x, the URL Filtering and Application Control blades enforce through the Access Control policy — typically a dedicated layer whose rules match applications and site categories, with actions Accept, Drop, Ask and Inform (UserCheck pages) and per-rule bandwidth limits, plus custom Applications/Sites objects for anything niche. What those rules can actually see is decided elsewhere: the HTTPS Inspection policy is its own rulebase of inspect/bypass rules, and on appliances it is commonly constrained or switched off for performance and complexity reasons — Check Point maintains a dedicated best-practice SK for it.

Export. The application/URL-filtering layers come out through the same mgmt_cli rulebase export; capture the HTTPS Inspection rulebase and its bypass list separately, and pull SmartLog/SmartEvent web reports as the evidence base for what each category action actually affects.

App Control / URL Filtering layer rules Internet Firewall rules with app and category conditions One rulebase governs apps and URLs — no blade licences per gateway
Check Point URL categories Cato URL categories Taxonomies differ in names and granularity — map by intent, review every blocked category individually
Ask / Inform actions (UserCheck pages) Prompt action User coaching — proceed after acknowledging the warning
Custom Applications/Sites & groups Custom apps and custom categories Consolidate allow/block lists and re-categorisations into custom categories referenced by rules
Access roles in web rules User/group-scoped Internet Firewall rules Depends on SCIM/LDAP group provisioning — sequence it first
Per-rule Limit action Bandwidth management rules Re-model the throttling intent, don't transliterate
HTTPS Inspection policy (inspect / bypass rules) Cato TLS Inspection — staged rebuild Never ported 1:1 — carry the bypass intent (banking/health categories, pinned apps), not the rules

Semantic gotchas. The taxonomy mismatch is the headline risk: a Check Point category and a similarly-named Cato category are not guaranteed to cover the same sites, so a mechanical name-for-name mapping silently changes what is blocked. The honest second: in many estates HTTPS Inspection never ran at full coverage, so the web policy has never been enforced against decrypted traffic — expect new detections as Cato TLSi stages in, and judge SWG parity only after decryption is staged. During co-existence, avoid double inspection: disable gateway HTTPS Inspection for Cato-routed traffic, or scope Cato TLSi bypasses to those paths until cutover — certificate-on-certificate failures break pinned apps twice over.

Strategy. Build a category translation sheet mapping each category-action to a Cato category and action by intent; consolidate custom Sites into custom categories; deploy the blocked set in monitor mode and compare events against SmartEvent web reports; rebuild TLS inspection through Cato's phased best practice — QUIC blocked and the root certificate deployed first, then gradual enablement from high-risk/low-impact categories outward, tuning bypasses as pinned apps surface — using Prompt where the customer used UserCheck pages.

Domain 3 — CASB policy

Inline CASB intent

The construct. There is no inline CASB blade in the Quantum rulebase to migrate. What exists is adjacent: Application Control signatures doing coarse SaaS gating in the access layers, and — where the customer bought it — Harmony SASE (the Perimeter 81-derived stack) with its own agent, its own Infinity Portal application and its own SaaS controls, in a policy plane entirely separate from SmartConsole. Expect a handful of configured controls at most: this is an intent-mapping exercise on top of a greenfield build, not an object migration.

Export. The Harmony SASE tenant configuration (if present) from its portal; any App Control rules touching SaaS applications from the access-policy export; SmartEvent application reporting as the more valuable artefact — it shows what the organisation actually uses, sanctioned or not.

Check Point constructCato constructNotes
Harmony SASE SaaS / web controls (if present) Cato CASB application rules Map the intent — different catalogue, different agent, different portal
App Control SaaS gating in access layers CASB granular activity rules Upload, download, share, log-in — per app and user group
Corporate-tenant enforcement intent Cato tenant control rules Corporate-tenant-only access to sanctioned suites
SmartEvent application reporting Cloud Apps Dashboard / App Analytics Shadow-IT discovery with app risk scoring, on by default

Semantic gotchas. The current state is genuinely split-brain — a Quantum rulebase and a Harmony SASE portal that share neither objects nor policy model — so inventory both planes before agreeing scope. Activity-level control on any platform requires TLS inspection: a CASB "migration" judged before decryption is staged will look falsely thin. And because the configured estate is usually near-empty, the real risk is anchoring scope to the old config rather than to discovered usage — migrate what the organisation needs, not the two controls it happened to have.

Strategy. Discovery first — let the Cloud Apps Dashboard build the real SaaS usage picture during the monitor phase, agree the sanctioned-app list with the customer, rebuild the few genuine controls as Cato CASB and tenant-control rules, then tighten from monitor to enforcement. Keep the Harmony SASE tenant warm until the final user cohort has migrated — it is a rollback lever, not day-one waste — and retire it with the last wave.

Domain 4 — DLP policy

The DLP blade

The construct. The Check Point DLP blade defines data types — keywords, dictionaries, regex patterns, file attributes and compound types — evaluated in rules with actions from Detect and Inform through Ask User to Prevent. In most estates it was licensed in a bundle and minimally deployed: Detect-only on a few gateways, bounded by the same HTTPS Inspection coverage that constrains the web blades. Frame this domain as greenfield-plus-migrate: build a data-protection programme on Cato DLP, and carry in the few custom patterns worth keeping.

Export. The DLP rulebase and data-type definitions from the management server. The custom regex patterns and keyword dictionaries are the only assets worth carrying forward — capture them verbatim.

Check Point constructCato constructNotes
Built-in data types (PCI · PII · financial) Cato predefined data types Prefer these over ported regexes — maintained by Cato, tuned for the engine
Custom patterns & dictionaries Custom data types Regex dialects differ — re-test every pattern against sample data
File attributes / compound data types File attribute conditions in DLP rules Combine with data types for precision
Detect / Inform actions DLP monitor mode The evidence-gathering phase — watched from the Data Protection Dashboard
Ask User (UserCheck coaching) Validate the requirement with Cato PS Map the intent in discovery rather than assuming parity
Prevent action Block — after monitor Enforcement earned per rule with event evidence
Repository fingerprints / whitelisted documents Exact Data Matching (EDM) Structured matching against customer-supplied datasets

Semantic gotchas. This is a rebuild, not a conversion — there is no importer, and that is the right outcome: a gateway-bound blade whose reach ended where HTTPS Inspection did does not map onto a cloud DLP engine that inspects everything the TLS policy decrypts. TLS inspection is the hard prerequisite; regex behaviour differs between engines; UserCheck-style Ask flows need their intent re-decided rather than assumed. And because blade deployments are typically thin, this migration is in practice an upgrade — from Detect-only rules on two gateways to an actual data-protection programme.

Strategy. Inventory the data types actually referenced by enforcing rules; recreate custom patterns as custom data types and swap built-ins for Cato's predefined types; run the whole policy in monitor with the Data Protection Dashboard as the evidence loop; tune false positives with the data owners; then enforce rule by rule.

What changed in translation

  • The hierarchy flattened — ordered and inline layers became one flat, first-match rulebase per firewall; every sub-rule now carries its effective scope explicitly instead of inheriting it from a parent rule.
  • Implicit cleanups became decisions — the inline layer's cleanup is now explicit rule 2, and the top-layer cleanup (rule 14) stops being a rule at all: east-west it maps to the WAN Firewall's implicit block, and for Internet egress it survives only as an explicit final block if the customer wants allowlist egress.
  • NAT travelled separately — nothing in either table touches NAT; the branch hide NAT disappears into PoP egress SNAT, and manual/static rules get their own line-by-line disposition in the site-level NAT policy, exactly as the FW-rules section describes.
  • Hit counts decided what got translated — the show-hits export ran before any of this: zero-hit sub-rules and dead layers exited at review, so the flattened rulebase is smaller than the layered one it replaces.
Cross-domain practices

Rules that hold across all four domains

🧭

Respect the default postures

Every Check Point layer carried its own implicit cleanup; Cato has two fixed postures — WAN Firewall allowlist (implicit block), Internet Firewall blocklist (implicit allow). Egress posture is therefore a decision, not a translation: if the customer wants allowlist egress, add an explicit final block rule. Don't discover the difference in production.

🔐

Stage TLS inspection early

HTTPS Inspection rarely ran at full coverage on the appliances, and SWG fidelity, CASB activity control and DLP content inspection all hang off decryption. Phased Cato TLSi — QUIC blocked, root certificate deployed in advance, category by category with pilot groups — is the fidelity prerequisite for three of the four domains.

🟠

Monitor, then block

Every translated rule, IPS/NGAM protection, CASB control and DLP rule starts in monitor. Enforcement is earned with event evidence, per rule — the pipeline's amber-to-green transition, never a big-bang flip.

🪪

SCIM first, then identity rules

Access roles ran on AD Query and Identity Collector servers — infrastructure Cato replaces with SCIM/LDAP sync, the Client and the Identity Agent. Provision the directory sync before policy deployment, and prove every identity-dependent rule on the Cato side before gateway-side identity sources are switched off.

🧱

Objects before rules

Export with the object dictionary, dedupe the network/host/service objects, and push the clean model through the GraphQL API or Terraform provider first — then build rules on top of it. Rules referencing ad-hoc objects recreate the sprawl you just removed.

📊

Validate with events & hit counts

The show-hits export is the baseline; Monitor → Events is the loop. Compare each translated rule's hits against its SmartConsole ancestor, investigate the deltas, and treat zero-hit rules after a full business cycle as candidates to delete — on the new platform as ruthlessly as on the old.

The policy pipeline

EXPORT → REVIEW & MAP → DEPLOY → OPTIMISE

The same pipeline runs for each domain, offset in time — firewall first, SWG behind it, CASB and DLP once TLS inspection and discovery data are in place. Dead rules and layer debt leave the pipeline at review and are never translated; everything that survives deploys in monitor and is promoted to enforcement on evidence.

Monitor first — log, don't block Enforce — promoted on evidence Discarded — dead rules never migrate
The four SmartConsole policy domains are exported once via the Management API, rationalised against hit-count evidence — zero-hit rules and layer debt exit the pipeline here — and only the surviving intent is mapped to CMA constructs. Every mapped rule deploys in monitor (amber) and is promoted to enforcement (green) when events and hit counts prove it behaves as intended.
