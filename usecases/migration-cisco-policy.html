Cisco Policy Migration

Moving a Cisco estate onto Cato is a routing project until the policy work begins — then it stalls, because ASA/FTD ACLs, Umbrella destination lists and web policies do not translate line-for-line into CMA. This page works through the four policy domains — firewall, SWG, CASB and DLP — with the export method for each, a construct-by-construct mapping, the semantic traps, and the four-phase EXPORT → REVIEW & MAP → DEPLOY → OPTIMISE process that turns a decade of accumulated rules into one clean rule base.

Business objective

Policy translation is where migrations stall

The network side of a Cisco-to-Cato migration is a routing event — reversible, well understood, covered on the Cisco to Cato page. The policy side is where programmes lose weeks: rules are scattered across ASDM/FMC/CDO and the Umbrella dashboard, the constructs carry different semantics, and nobody can say which of the accumulated exceptions still matter. The temptation is to copy everything across “to be safe” — which imports ten years of dead rules into a brand-new platform and makes the Cato rule base unauditable on day one.

The Cato PS approach is a four-phase pipeline applied per policy domain: EXPORT the vendor policy in a reviewable form → REVIEW & MAP — rationalise on hit counts and usage evidence first, then map only the survivors to CMA constructs → DEPLOY everything monitor-first, so rules log but do not block → OPTIMISE — tighten to enforcement using the events and hit counts the monitor phase produced. Never lift-and-shift: rule bases shrink substantially when separate DNS, web and firewall policies collapse into one inspection engine at the PoP.

Provenance — read before you present

This page is authored from vendor documentation and Cato PS methodology research; it is not a delivery-verified runbook. Review the mappings and the monitor-to-enforce criteria with Cato Professional Services before committing them to a customer plan.

Ask yourself

“What percentage of your ASA ACEs show a zero hit count today — and how many Umbrella destination-list entries would anyone in the room still defend?”

Domain 1 · FW rules

ASA/FTD access policy & Umbrella cloud firewall

ASA and FTD policy lives in interface ACLs built from ACEs and object-groups (network, service, protocol), managed through ASDM, FMC or CDO; FTD adds a Snort-based IPS policy on top. Umbrella SIG contributes a second, easily forgotten firewall: the cloud-delivered firewall (CDFW) with L3/L4 rules that apply only to traffic tunnelled into Umbrella.

Export: on ASA, show running-config for the objects and show access-list for the expanded ACEs — the latter includes a live hitcnt per entry, which is the single most valuable artefact in the whole exercise. FMC exports access policies and reports rule hit counts; CDO holds the same data for cloud-managed devices. Capture the Umbrella CDFW rule set from the dashboard for review alongside them.

Cisco constructCato constructMigration note
Interface ACLs / ACEs (ASA, FTD access policy)WAN firewall + Internet firewall ordered rulesSplit by direction: site-to-site and east-west into the WAN firewall (allowlist), outbound exceptions into the Internet firewall (blocklist). One global base, not per-interface.
Network object-groupsHosts, IP ranges and groups in CMAFlatten nested groups; delete unreferenced objects during clean-up rather than importing them.
Service / protocol object-groupsServices and custom services (protocol/port)Many collapse into Cato’s built-in application and service catalogue.
Interfaces, zones, security levelsSite / network-range scoping on rulesZones do not exist in Cato — scope rules by site, VLAN range or group instead.
Umbrella CDFW L3/L4 rulesInternet firewall rulesCDFW only ever saw tunnelled traffic; in Cato the same base applies to every site and user flow.
FTD Snort IPS policyCato IPS (managed)No signature tuning carries over — Cato IPS is managed by Cato. Run it in monitor before prevent.
NAT and port-forward rulesPoP egress NAT · remote port forwarding on allocated IPsEgress NAT policy largely disappears; inventory inbound port-forwards separately and rebuild deliberately.

Semantic gotchas

  • Direction inversion. An interface ACL ends in an implicit deny; Cato splits postures — WAN firewall allowlist, Internet firewall blocklist. A raw copy puts outbound exceptions into the wrong base and inverts their meaning.
  • Shadowing on merge. Both models are first-match, but consolidating several per-interface ACLs into one ordered base can make a rule that never matched on ASA shadow a rule that did. Re-order deliberately.
  • Object sprawl. Nested object-groups hide dead entries; expand them (show access-list does this for you) before deciding what survives.
  • NAT does not travel. Twice-NAT and object NAT have no Cato equivalent to translate into — the PoP handles egress NAT, and inbound publishing becomes remote port forwarding on allocated IPs.

Migration strategy

Export with hit counts → cull zero-hit ACEs (with change-control sign-off — some are annual DR paths, most are dead) → flatten the object-groups → sort survivors into WAN firewall (allowlist) versus Internet firewall (exceptions to the blocklist) → deploy with rules set to allow-and-track or monitor first, and tighten to block from the event evidence.

Domain 2 · SWG policy

Umbrella web policies, destination lists & decryption

Umbrella’s secure web gateway policy spans DNS policies and (with SIG) web policies, built from destination lists (allow/block), content category settings and selective decryption lists — all evaluated against an identity-precedence model (roaming client > AD user/group > internal network > network).

Export: destination lists download to CSV from the dashboard or via the Destination Lists API; category settings and rule structure come from a policy walk-through, and the reporting views show which categories and destinations actually fire — the SWG equivalent of firewall hit counts.

Umbrella constructCato constructMigration note
Destination lists (block)Custom categories + Internet firewall block rulesDedupe against Cato system categories first — many entries exist only because a category miss was patched by hand years ago.
Destination lists (allow)Allow rules / category exceptionsCheck each allow for a TLS-inspection bypass need at the same time.
Content category settingsInternet firewall category rulesTaxonomies differ — map by intent, never by name, and re-baseline in monitor mode.
DNS-layer enforcementCato DNS Security / DNS Protections (IPS)Inline at the PoP for all DNS traffic, whatever resolver the client uses.
Selective decryption listsTLS Inspection bypass rulesRebuild deliberately rather than transcribing — the exemption logic is expressed differently.
Ruleset identities (precedence model)Ordered rules with sites, users and groups as sourcesImplicit precedence becomes explicit rule order — the biggest semantic shift on this page.
Block-page bypass codesNo direct equivalentAgree an override workflow (e.g. time-boxed allow rules) during Review & Map.
Per-site policy moves at swap time — not later

Umbrella network identities are keyed to the site’s public egress IP. The moment a Socket goes live, the site egresses from a shared Cato PoP address and stops matching its Umbrella policy entirely. Migrate each site’s Umbrella web/DNS policy to Cato in the same change window as the socket swap — it is not a follow-up task, it is part of the cutover.

Semantic gotchas

  • Category taxonomy mismatch. An Umbrella category can span two Cato categories, or the reverse. Run the mapped rules in monitor and compare what fires against Umbrella’s reports before claiming parity.
  • SSL decryption is a re-build, not a port. The Umbrella root CA on endpoints is replaced by the Cato certificate (push via MDM), and the decryption scope is re-expressed as TLS Inspection rules with bypasses for pinned apps. Do this early — it gates SWG, CASB and DLP fidelity alike.
  • Precedence to order. An Umbrella rule that “won” through identity precedence may need to move up or down the Cato base to keep the same effective behaviour — walk the top-20 destinations per user population to confirm.
  • Niche behaviours. Bypass codes, per-identity block pages and similar features have no 1:1 equivalent — verify each during Review & Map instead of assuming.

Migration strategy

Export the destination lists → dedupe against Cato system categories and drop what a category rule already covers → build custom categories for the remainder → recreate the DNS-layer controls in DNS Protections → deploy the whole set in monitor → move each site’s policy at socket-swap time → confirm parity from events, then enforce.

Domain 3 · CASB policy

Umbrella app discovery & blocking → App Control

Umbrella’s CASB footprint is usually modest: App Discovery (cloud-app usage inferred from DNS and proxy logs) plus application block settings inside the web policy. There is rarely a deep rule base to convert — so this domain is a mapping of intent, not of rules: which apps are blocked, tolerated or sanctioned, and for whom. Export: the App Discovery report to CSV, and the application settings from a policy walk-through.

Umbrella constructCato constructMigration note
App Discovery reportsCloud Apps Dashboard / shadow-IT analyticsRun a Cato discovery baseline in parallel and compare the two before enforcing anything.
Application block settingsApp Control (CASB) rules + Internet firewall app rulesUmbrella blocks whole apps at the domain level; Cato adds granular activities (upload, download, post).
App risk informationCato app risk scores in App AnalyticsRe-tag sanctioned vs unsanctioned in CMA — labels do not carry over.

Semantic gotchas

  • Domain-level vs activity-level. “Block Dropbox” in Umbrella becomes a choice in Cato: block the app, or allow it read-only by blocking uploads. Decide per app rather than defaulting to the blunt option.
  • App catalogues differ. Names and app groupings will not match one-for-one — map by function and verify with a test flow.
  • Granular control needs TLS inspection. Activity-level CASB rules on HTTPS traffic require the inspection scope from the SWG domain to be in place first.

Migration strategy

Extract the intent (blocked / tolerated / sanctioned app lists per user population) → re-express as App Control rules with activities where useful → run Cato’s own discovery for a baseline period and reconcile against Umbrella App Discovery → monitor, review the deltas with the customer, then enforce.

Domain 4 · DLP policy

Umbrella real-time DLP → Cato DLP profiles

Umbrella real-time DLP (SIG Advantage) inspects traffic inline against data classifications — built-in identifiers plus custom regex patterns and dictionaries — in rules scoped to identities and destinations. Most estates enforce only a handful of rules, so treat this domain like CASB: small, but semantically fiddly. Export: the classification inventory and rule set from a policy walk-through, plus incident reports for tuning context — there is no one-click policy export.

Umbrella constructCato constructMigration note
Built-in data classificationsCato DLP content profiles — predefined data typesCoverage differs by type (PCI, PII, credentials…) — validate each classification you actually use.
Custom regex / dictionariesCustom data types in DLP content profilesPort the patterns, then re-test thresholds — match confidence is engine-specific.
Real-time DLP rules (identities × destinations)DLP rules scoped by user/group, app or categoryRequires TLS inspection on the traffic in scope — sequence after the SWG decryption re-build.
DLP incident reportsEvents + Data Protection DashboardUse the last 90 days of Umbrella incidents as the tuning corpus for monitor mode.

Semantic gotchas

  • Thresholds do not transfer. Match counts and confidence levels behave differently per engine — expect a tuning cycle in monitor mode, not a like-for-like flip.
  • File-type coverage differs. Confirm the formats that matter to the customer (archives, images with OCR expectations, CAD) rather than assuming parity.
  • Inline-first. Cato DLP enforcement is inline at the PoP; if the customer relied on out-of-band SaaS API scanning, verify that requirement’s scope with Cato PS during Review & Map.

Migration strategy

Rebuild only the rules Umbrella actually enforced → recreate the classifications as content profiles (predefined types first, custom regex second) → run in monitor with events feeding the Data Protection Dashboard → tune thresholds against the Umbrella incident corpus → then, and only then, switch to block.

What changed in translation

  • Identity is real, not inferred. Umbrella rows 2–3 matched network identities keyed to the sites’ public egress IPs — meaningless the moment those sites egress from a Cato PoP. The Cato rules match SCIM-provisioned user groups and site objects directly.
  • Warn becomes Prompt. Umbrella’s Warn interstitial maps to Cato’s Prompt action — the user can proceed and the event is logged. Re-baseline the category match in monitor first: the taxonomies differ.
  • Hit counts culled the ACL. Two of three ACEs survive; the zero-hit deny is dropped at Rationalise (with change-control sign-off), not imported “to be safe”.
  • Object-groups become CMA objects. BRANCH-NETS and DC-APP-SERVERS flatten into hosts, ranges and groups — expand nested groups first (show access-list does this for you) so dead entries are visible before translation.
Cross-domain practice

Rules that hold across all four domains

🧭

Respect the default postures

Cato’s WAN firewall is an allowlist; the Internet firewall is a blocklist. Sort every legacy rule into the right base — copying an ACL wholesale inverts semantics.

🔍

TLS inspection is the fidelity prerequisite

Category matching, App Control activities and DLP are all blind on encrypted traffic without it. Push the Cato certificate early via MDM and stage inspection with bypasses for pinned apps.

🟡

Monitor, then block

Every imported rule starts life logging, not blocking. Flip to enforce only on event evidence — per rule, not per big-bang.

🪪

Identity before policy

Provision IdP SSO and SCIM groups before importing any rule that references users. Umbrella AD identities map onto those groups — and group-name drift breaks rules silently.

📊

Validate with events and hit counts

Parity is proven, not asserted: compare Umbrella and FMC reports against CMA events and rule hits per wave before decommissioning anything.

⏱

Web policy moves at swap time

Umbrella network identities break the moment a site egresses via a Cato PoP — per-site policy migration is part of the socket cutover window, never a follow-up.
